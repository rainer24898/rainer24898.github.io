# CPU Rings (Protection Rings) và cách User Space tương tác với Driver

## 1. Mở đầu

Khi học hệ điều hành, kernel, driver, hoặc debug chương trình C/Linux ở mức thấp, có một khái niệm gần như bắt buộc phải nắm: **CPU rings** hay **protection rings**.

Đây là cơ chế phần cứng dùng để phân cấp **mức đặc quyền** khi thực thi mã lệnh. Nhờ đó, CPU và hệ điều hành có thể tách biệt rõ:

* mã **user space**
* mã **kernel space**
* các thao tác được phép và không được phép
* vùng nhớ nào được truy cập trực tiếp
* đường đi hợp lệ khi muốn chạm vào tài nguyên nhạy cảm

Nếu không có cơ chế này, bất kỳ chương trình user nào cũng có thể tự sửa page table, đụng vào bộ nhớ kernel, điều khiển thiết bị trực tiếp, hoặc làm sập toàn bộ hệ thống chỉ bằng một bug nhỏ.

---

## 2. CPU Rings là gì?

Trong kiến trúc **x86**, mô hình protection truyền thống có **4 mức đặc quyền**:

* **Ring 0**: đặc quyền cao nhất
* **Ring 1**
* **Ring 2**
* **Ring 3**: đặc quyền thấp nhất

Số ring **càng nhỏ** thì quyền **càng cao**.

### Ý nghĩa cốt lõi

Ring quyết định:

* đoạn mã đó có được chạy lệnh đặc quyền hay không
* có được truy cập một số tài nguyên nhạy cảm hay không
* CPU có cho phép thao tác đó diễn ra hay không

Nói ngắn gọn:

* **Ring 0**: gần với quyền kiểm soát hệ thống nhất
* **Ring 3**: chỉ là mã ứng dụng thông thường, bị giới hạn mạnh

---

## 3. Các ring trong thực tế được dùng như thế nào?

### Ring 0 — Kernel mode

Đây là mức đặc quyền cao nhất. Kernel của hệ điều hành thường chạy ở đây.

Ở Ring 0, mã lệnh có thể:

* thực thi các **privileged instructions**
* truy cập trực tiếp phần cứng
* quản lý page table
* xử lý interrupt/exception
* chạy scheduler
* thao tác MMIO, DMA, APIC, I/O port
* truy cập bộ nhớ kernel

Nếu có bug ở đây, hậu quả rất nặng vì nó có thể:

* crash kernel
* corrupt bộ nhớ toàn hệ thống
* làm treo máy
* tạo lỗ hổng bảo mật nghiêm trọng

### Ring 1 và Ring 2

Về mặt kiến trúc x86, Ring 1 và Ring 2 **có tồn tại**.

Tuy nhiên trong phần lớn hệ điều hành hiện đại như Linux và Windows, hai ring này **hầu như không được dùng theo kiểu phân tầng driver ở giữa kernel và user** như sách vở hay mô tả lý thuyết.

Cách dùng phổ biến hơn là:

* **kernel + kernel-mode driver** chạy ở **Ring 0**
* **ứng dụng user-space** chạy ở **Ring 3**

Tức là trên thực tế, nhiều hệ điều hành sử dụng protection model gần giống **2 mức chính**:

* kernel mode
* user mode

### Ring 3 — User mode

Đây là mức đặc quyền thấp nhất trong mô hình 4 ring của x86.

Hầu hết mọi ứng dụng thông thường chạy ở đây:

* shell
* trình duyệt
* editor
* service user-space
* chương trình C/C++ bình thường
* game
* tool debug user-space

Ở Ring 3, chương trình **không được phép**:

* sửa thanh ghi điều khiển như `CR0`, `CR3`, `CR4`
* tắt interrupt bằng `cli`
* thao tác I/O port tùy ý
* đụng trực tiếp page table
* truy cập bộ nhớ kernel
* điều khiển phần cứng trực tiếp theo ý muốn

Muốn làm những việc đó, nó phải đi qua **kernel** bằng **system call** hoặc một cơ chế đã được hệ điều hành cho phép.

---

## 4. Ring là khái niệm phần cứng, còn user mode / kernel mode là cách hệ điều hành dùng nó

Đây là điểm rất quan trọng.

* **Protection ring** là khái niệm ở mức **CPU architecture**
* **Kernel mode / user mode** là cách **hệ điều hành ánh xạ và sử dụng** cơ chế đó

Vì vậy khi học, nên tách làm 2 lớp:

### Góc nhìn phần cứng

CPU có:

* privilege level
* privileged instruction
* interrupt / exception / trap
* segment/ring checks
* page permission checks

### Góc nhìn hệ điều hành

OS tổ chức:

* user process ở mức thấp
* kernel ở mức cao
* syscall là cổng vào hợp lệ
* driver được kernel điều phối
* tài nguyên được quản lý tập trung

---

## 5. Vì sao phải tách Ring 0 và Ring 3?

### 5.1. Bảo mật

Nếu một tiến trình user có đặc quyền như kernel, nó có thể:

* đọc trộm bộ nhớ tiến trình khác
* sửa kernel code/data
* thao tác driver và thiết bị trực tiếp
* chiếm quyền toàn hệ thống

### 5.2. Ổn định

Bug trong ứng dụng user mode thường chỉ làm chết chính tiến trình đó.

Bug trong kernel mode có thể làm:

* kernel panic
* BSOD
* treo toàn bộ hệ thống
* hỏng cấu trúc quản lý bộ nhớ

### 5.3. Kiểm soát tài nguyên tập trung

Kernel là lớp trung gian đứng giữa mọi ứng dụng và tài nguyên hệ thống:

* CPU scheduling
* memory management
* file system
* network stack
* IPC
* driver
* security policy

Nhờ đó, hệ điều hành có thể kiểm soát ai được làm gì.

---

## 6. Ring switch là gì?

**Ring switch** là việc CPU chuyển mức đặc quyền khi thực thi mã.

Ví dụ điển hình:

* từ **Ring 3** sang **Ring 0** khi gọi system call
* từ **Ring 0** về **Ring 3** khi system call hoàn tất

Đây là **chuyển privilege level**, không nhất thiết là đổi sang một process hay thread khác.

Ví dụ:

```c
ssize_t n = read(fd, buf, sizeof(buf));
```

Luồng có thể là:

1. thread A đang chạy ở Ring 3
2. gọi `read()`
3. CPU vào kernel ở Ring 0
4. kernel xử lý syscall cho chính thread A
5. quay về Ring 3

Trong trường hợp này có **ring switch**, nhưng **không bắt buộc có context switch**.

---

## 7. Context switch là gì?

**Context switch** là việc CPU/người lập lịch đổi từ thread/process đang chạy sang thread/process khác.

Ví dụ:

* thread A đang chạy
* scheduler quyết định chuyển CPU sang thread B
* phải lưu/khôi phục register context, stack pointer, scheduling state, memory context tương ứng

### Context thường bao gồm

* general-purpose registers
* instruction pointer
* stack pointer
* flags
* scheduling state
* memory mapping liên quan
* kernel stack của thread

### Điểm cần nhớ

* **Ring switch**: đổi mức đặc quyền
* **Context switch**: đổi thực thể đang chạy

Hai khái niệm này **không giống nhau**.

---

## 8. Phân biệt ring switch và context switch

| Tiêu chí                                 | Ring switch                    | Context switch                          |
| ---------------------------------------- | ------------------------------ | --------------------------------------- |
| Bản chất                                 | Đổi mức đặc quyền              | Đổi thread/process đang chạy            |
| Ví dụ                                    | `syscall` từ Ring 3 vào Ring 0 | scheduler đổi từ thread A sang thread B |
| Có cần đổi thread không?                 | Không bắt buộc                 | Có                                      |
| Có thể xảy ra một mình không?            | Có                             | Có                                      |
| Liên quan trực tiếp đến scheduler không? | Không nhất thiết               | Có                                      |
| Mục đích                                 | vào vùng mã đặc quyền          | chia CPU cho thực thể khác              |

### Trường hợp 1: chỉ có ring switch

```text
Thread A (Ring 3) -> syscall -> Thread A (Ring 0) -> return -> Thread A (Ring 3)
```

Không đổi thread, chỉ đổi privilege level.

### Trường hợp 2: có cả ring switch và context switch

```text
Thread A (Ring 3)
   -> syscall vào kernel (Ring 0)
   -> bị block chờ I/O
   -> scheduler chuyển sang Thread B
```

Lúc này:

* lúc vào kernel có **ring switch**
* khi scheduler đổi A sang B có **context switch**

---

## 9. System call là cổng đi hợp lệ từ Ring 3 vào Ring 0

Ứng dụng user-space không thể tùy tiện nhảy vào mã kernel chỉ bằng cách gọi hàm như gọi hàm C bình thường.

Muốn yêu cầu kernel làm việc đặc quyền, ứng dụng phải dùng **system call**.

Ví dụ trong Linux:

* `read()`
* `write()`
* `open()`
* `ioctl()`
* `mmap()`
* `socket()`

Ví dụ trong Windows:

* `ReadFile()`
* `WriteFile()`
* `DeviceIoControl()`

### Ý tưởng tổng quát

1. User code đang chạy ở Ring 3
2. Gọi wrapper trong libc hoặc API tương ứng
3. Wrapper đặt syscall number và các tham số vào register theo ABI
4. CPU thực thi lệnh như `syscall`, `sysenter`, hoặc cơ chế cũ kiểu software interrupt
5. CPU chuyển vào **kernel entry path**
6. Kernel chạy ở Ring 0 để xử lý yêu cầu
7. Kết thúc thì quay lại user mode

---

## 10. Syscall entry nhìn theo góc độ học kernel

Trên x86-64 Linux, khi user-space gọi một syscall, bức tranh logic thường là:

```text
User code
  -> libc wrapper
      -> instruction: syscall
          -> CPU chuyển sang kernel entry
              -> kernel save context cần thiết
              -> kernel dùng kernel stack
              -> tra syscall number
              -> gọi hàm xử lý syscall
              -> trả kết quả
          -> sysret/iretq
  -> quay lại user code
```

### CPU thường phải làm những gì khi vào kernel

* đổi mức đặc quyền từ 3 xuống 0
* chuyển sang địa chỉ entry do kernel thiết lập
* dùng **kernel stack** thay vì user stack
* lưu đủ trạng thái để sau này quay lại user mode

### Điểm rất quan trọng

**System call không đồng nghĩa với context switch.**

Phần lớn thời gian, chính thread đang chạy chỉ tạm đi vào kernel, xử lý xong rồi quay lại.

---

## 11. Vì sao khi vào kernel lại dùng kernel stack?

Mỗi thread thường có:

* **user stack** dùng khi chạy ở user mode
* **kernel stack** dùng khi đi vào kernel mode

Lý do không dùng tiếp user stack trong kernel:

* user stack nằm trong vùng user memory, không đáng tin tuyệt đối
* kernel cần vùng stack riêng, an toàn hơn
* dễ quản lý context khi xử lý interrupt/syscall/exception

### Hình dung đơn giản

```text
Thread A:

User mode:
    RIP -> user code
    RSP -> user stack

Syscall vào kernel:
    RIP -> kernel entry
    RSP -> kernel stack của thread A
```

## 12. Page table và vì sao ring thôi là chưa đủ

Rất nhiều người mới học dễ nghĩ:

> User không đụng được kernel memory chỉ vì nó chạy ở Ring 3.

Cách hiểu này **đúng nhưng chưa đủ**.

Trong hệ thống hiện đại, bảo vệ bộ nhớ còn phụ thuộc mạnh vào:

* **virtual memory**
* **page table**
* bit **U/S (User/Supervisor)**
* bit **R/W**
* bit **NX**
* cơ chế như **SMEP**, **SMAP**, **KPTI**

### Ring quyết định cái gì?

Ring quyết định **mức đặc quyền của mã đang chạy**.

### Page table quyết định cái gì?

Page table quyết định **địa chỉ ảo nào được map đến đâu** và **được phép truy cập kiểu gì**.

Hai thứ này phối hợp với nhau để bảo vệ hệ thống.

---

## 13. U/S bit là gì?

Trong page table của x86, mỗi page entry có các bit quyền quan trọng. Một trong số đó là:

* **U/S bit** = User / Supervisor

Ý nghĩa đơn giản:

* **U/S = User**: page có thể được truy cập từ user mode nếu các điều kiện khác cũng hợp lệ
* **U/S = Supervisor**: page chỉ được truy cập khi đang ở mức đặc quyền supervisor/kernel

### Tại sao bit này quan trọng?

Vì nó giúp CPU kiểm tra:

* code đang chạy ở Ring 3 có được đọc/ghi page này không
* hay page đó chỉ dành cho kernel

Nếu user mode cố truy cập vào page supervisor-only, CPU sẽ sinh fault.

---

## 14. Hình dung đơn giản về virtual address, page table và U/S bit

```text
User process tạo địa chỉ ảo
        |
        v
+---------------------+
| Virtual Address     |
+---------------------+
        |
        v
+---------------------+
| Page Table Walk     |
| kiểm tra PTE        |
| - Present?          |

```

### Nếu user code ở Ring 3 truy cập kernel page

```text
CPU thấy:
- current privilege: user
- page: supervisor only
=> từ chối truy cập
=> fault
```

## 15. Ring + page permission phối hợp với nhau như thế nào?

Hãy xem 2 ý này như hai lớp bảo vệ:

### Lớp 1: privilege level

CPU biết mã hiện tại đang chạy ở:

* Ring 0
* hay Ring 3

### Lớp 2: permission của page

CPU biết page đó có thuộc loại:

* user-accessible
* supervisor-only
* read-only
* non-executable

### Kết quả

Một truy cập bộ nhớ chỉ thành công khi:

* mức đặc quyền phù hợp
* page permission cho phép
* loại thao tác phù hợp: đọc/ghi/thực thi

Tức là **ring không tự bảo vệ mọi thứ**, mà phải kết hợp với **page permission**.

---

## 16. Luồng đầy đủ: User app -> Kernel -> Driver -> Hardware -> User app

Bây giờ ghép mọi thứ lại thành một flow hoàn chỉnh.

### Bước 1: User app tạo yêu cầu ở Ring 3

Ví dụ chương trình muốn đọc dữ liệu từ thiết bị hoặc điều khiển thiết bị:

```c
int fd = open("/dev/mydev", O_RDWR);
ioctl(fd, MYDEV_DO_SOMETHING, &arg);
read(fd, buf, 128);
```

Ở đây, chương trình **không trực tiếp thao tác phần cứng**. Nó chỉ gọi API do hệ điều hành cung cấp.

---

### Bước 2: libc wrapper kích hoạt system call

`read()` hay `ioctl()` trong user-space thực ra chỉ là lớp bọc để đi vào kernel.

Trên x86-64 Linux, wrapper cuối cùng sẽ dẫn đến lệnh kiểu `syscall`.

Lúc này CPU sẽ:

* đổi privilege level từ Ring 3 sang Ring 0
* nhảy vào kernel entry path
* dùng kernel stack
* lưu trạng thái cần thiết để quay lại user mode

---

### Bước 3: Kernel vào syscall entry và tra syscall table

Kernel nhận được:

* syscall number
* các tham số trong register theo ABI

Sau đó kernel sẽ:

* tra **syscall table**
* tìm hàm xử lý tương ứng
* validate tham số
* kiểm tra con trỏ user
* kiểm tra quyền truy cập
* resolve object kernel tương ứng với `fd`

Ví dụ với `read(fd, ...)`, kernel phải tìm ra object file tương ứng rồi đi qua các lớp như VFS/device layer trước khi tới chỗ xử lý phù hợp.

---

### Bước 4: Kernel subsystem hoặc driver xử lý

Đây là điểm cần hiểu đúng:

**không phải mọi syscall đều đi thẳng vào driver ngay lập tức**.

Tùy loại tài nguyên, luồng có thể đi qua nhiều lớp trung gian:

* VFS
* filesystem
* block layer
* network stack
* socket layer
* device framework
* actual driver

Ví dụ:

* `read()` trên file có thể đi qua VFS + filesystem + block layer + storage driver
* `send()` có thể đi qua socket layer + TCP/IP stack + NIC driver
* `ioctl()` trên `/dev/mydev` có thể đi tương đối trực tiếp vào callback của driver character device

---

### Bước 5: Driver thao tác với hardware

Nếu tới driver kernel-mode, driver có thể:

* đọc/ghi register của thiết bị
* truy cập MMIO
* dùng DMA
* bật bit start
* poll trạng thái
* xử lý interrupt khi hardware hoàn tất

Ví dụ thiết bị có thể là:

* NIC
* SSD
* UART
* GPU
* PCIe accelerator
* IP block trên SoC

Đây là phần user-space bình thường không được làm trực tiếp.

---

### Bước 6: Driver trả kết quả cho kernel

Sau khi hardware hoặc driver logic xử lý xong, kết quả được trả lại cho kernel.

Nếu cần copy dữ liệu sang user-space, kernel sẽ dùng các API copy an toàn, ví dụ Linux có:

* `copy_to_user()`
* `copy_from_user()`

Điểm quan trọng:

* kernel **không tin trực tiếp con trỏ user**
* luôn cần validate/copy đúng cách
* tránh để user ép kernel đọc/ghi lung tung vào vùng nhớ nguy hiểm

---

### Bước 7: Kernel trả về Ring 3

Khi syscall hoàn tất:

* kernel đặt giá trị trả về
* khôi phục user context
* CPU dùng cơ chế return thích hợp để quay lại user mode

Lúc này chương trình tiếp tục chạy ở dòng sau syscall.

## 19. Ví dụ C: `read()` trong user space

### User-space code

```c
#include <fcntl.h>
#include <stdio.h>
#include <unistd.h>

int main(void)
{
    int fd = open("data.bin", O_RDONLY);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    char buf[64];
    ssize_t n = read(fd, buf, sizeof(buf));
    if (n < 0) {
        perror("read");
        close(fd);
        return 1;
    }

    printf("read %zd bytes
", n);
    close(fd);
    return 0;
}
```

### Bên dưới có thể diễn ra gì?

```text
read(fd, buf, 64)
   -> libc wrapper
   -> syscall vào kernel
   -> kernel tra fd
   -> VFS xử lý
   -> filesystem xử lý
   -> block layer xử lý nếu cần
   -> storage driver / device
   -> dữ liệu được đưa về kernel buffer/page cache
   -> copy_to_user(buf, ...)
   -> return về user
```

### Ý nghĩa cần nhớ

`read()` nhìn rất đơn giản trong C, nhưng phía dưới thường là một chuỗi layer khá dài.

## 20. Ví dụ C: `ioctl()` để điều khiển device

`ioctl()` thường được dùng khi driver cần giao diện điều khiển riêng ngoài các thao tác đọc/ghi thông thường.

### User-space code

```c
#include <fcntl.h>
#include <stdio.h>
#include <sys/ioctl.h>
#include <unistd.h>

#define MY_IOCTL_SET_VALUE  _IOW('M', 1, int)
#define MY_IOCTL_GET_VALUE  _IOR('M', 2, int)

int main(void)
{
    int fd = open("/dev/mydev", O_RDWR);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    int value = 42;

    if (ioctl(fd, MY_IOCTL_SET_VALUE, &value) < 0) {
        perror("ioctl SET");
        close(fd);
        return 1;
    }

    value = 0;
    if (ioctl(fd, MY_IOCTL_GET_VALUE, &value) < 0) {
        perror("ioctl GET");
        close(fd);
        return 1;
    }

    printf("device value = %d ", value);
    close(fd);
    return 0;
}
```

### Driver-side logic thường là gì?

Về mặt ý tưởng, driver kernel có thể đăng ký callback kiểu:

```c
static long mydev_ioctl(struct file *file, unsigned int cmd, unsigned long arg)
{
    int value;

    switch (cmd) {
    case MY_IOCTL_SET_VALUE:
        if (copy_from_user(&value, (int *)arg, sizeof(value))) {
            return -1;
        }

        /* ghi value xuống device register hoặc lưu trạng thái nội bộ */
        return 0;

    case MY_IOCTL_GET_VALUE:
        value = 123; /* ví dụ đọc từ hardware hoặc state trong driver */

        if (copy_to_user((int *)arg, &value, sizeof(value))) {
            return -1;
        }
        return 0;

    default:
        return -1;
    }
}
```

> Đây là ví dụ minh họa ý tưởng, không phải đoạn driver production-ready hoàn chỉnh.

### Luồng logic

```text
User ioctl(fd, CMD, &value)
    -> syscall vào kernel
    -> kernel xác định file_operations của device
    -> gọi callback ioctl của driver
    -> driver copy dữ liệu từ user vào kernel
    -> driver thao tác hardware / state
    -> driver copy kết quả về user nếu cần
    -> return về user mode
```

## 21. Điều gì xảy ra nếu user cố làm việc đặc quyền trực tiếp?

Nếu code ở Ring 3 cố thực hiện thao tác bị cấm, CPU có thể từ chối bằng exception/fault.

Ví dụ các thao tác nhạy cảm:

* sửa `CR3`
* tắt interrupt bằng `cli`
* thao tác I/O port không được phép
* truy cập page supervisor-only
* thực thi từ vùng nhớ bị đánh dấu non-executable

Hậu quả tùy trường hợp có thể là:

* general protection fault
* page fault
* segmentation fault
* illegal instruction / privileged instruction fault
* tiến trình bị kernel kill

Điều này cho thấy protection không chỉ là “OS quy định”, mà còn được **CPU cưỡng chế ở mức phần cứng**.

---

## 22. Driver có luôn chạy ở Ring 0 không?

Không phải lúc nào cũng vậy.

Có hai kiểu nhìn phổ biến:

### 22.1. Kernel-mode driver

* chạy với đặc quyền kernel
* trong bối cảnh x86 truyền thống, thường tương ứng Ring 0
* hiệu năng cao
* thao tác phần cứng trực tiếp tốt hơn
* nhưng bug rất nguy hiểm

### 22.2. User-mode driver

* một phần logic ở user-space
* an toàn hơn
* crash ít kéo sập cả hệ thống
* nhưng thường cần lớp trung gian kernel để tiếp cận phần cứng

Trong Linux truyền thống, phần lớn driver chuẩn vẫn là kernel-mode driver.

## 23. Ring -1 trong ảo hóa là gì?

Khi nói về virtualization, tài liệu thường dùng cách nói **Ring -1** để chỉ mức đặc quyền của **hypervisor**.

Đây là cách nói trực quan để diễn tả rằng:

* guest OS có thể nghĩ nó đang chạy kernel ở Ring 0
* nhưng hypervisor mới là lớp thật sự kiểm soát tài nguyên máy vật lý

Tuy nhiên cần hiểu đúng:

* **Ring -1** không đơn giản chỉ là “thêm một ring mới” vào mô hình cổ điển
* nó là cách nói cho cơ chế đặc quyền do **hardware virtualization** hỗ trợ
* trên Intel là logic gắn với **VMX root / non-root**
* trên AMD là cơ chế tương tự của SVM

## 24. Một số câu rất dễ nhầm khi mới học

### Nhầm 1: syscall là context switch

Sai. Syscall trước hết là **ring switch**. Nó chỉ trở thành context switch nếu scheduler đổi thực thể đang chạy.

### Nhầm 2: driver luôn là lớp kế ngay sau syscall

Không hẳn. Nhiều syscall đi qua rất nhiều lớp như VFS, filesystem, socket layer, network stack, block layer.

### Nhầm 3: chỉ cần ring là đủ bảo vệ bộ nhớ

Sai. Bảo vệ bộ nhớ cần cả:

* ring
* page table
* U/S bit
* R/W bit
* NX bit
* các cơ chế bảo vệ khác

### Nhầm 4: kernel đang chạy là đang đổi sang process khác

Không đúng. Rất thường xuyên vẫn là **chính thread đó**, chỉ tạm đi vào kernel mode để xử lý syscall.

## 25. Góc nhìn dành cho lập trình viên C/Linux

Khi viết chương trình C bình thường:

* code của bạn chạy ở **Ring 3**
* nhiều API như `read`, `write`, `mmap`, `socket`, `ioctl` cuối cùng đi qua kernel
* bạn không được tự chạm vào tài nguyên đặc quyền như kernel làm

Khi viết kernel module/driver:

* code chạy với đặc quyền kernel
* mọi bug con trỏ, race condition, use-after-free, stack corruption đều nguy hiểm hơn rất nhiều
* lỗi có thể không chỉ làm chết tiến trình, mà làm hỏng cả hệ thống

Đó là lý do debug kernel khó hơn debug user-space rất nhiều.

## 27. Kết luận

CPU rings là cơ chế nền tảng để tách biệt **mã ít đặc quyền** và **mã đặc quyền cao**.

Trong hệ điều hành hiện đại, cách dùng phổ biến nhất là:

* **Ring 0** cho kernel và phần lớn kernel-mode driver
* **Ring 3** cho ứng dụng user-space

Khi ứng dụng cần làm việc đặc quyền, nó phải đi qua **system call** để vào kernel.

Nhưng để hiểu bảo vệ hệ thống một cách đúng bản chất, cần nhớ thêm 3 ý rất quan trọng:

1. **Ring switch** khác với **context switch**
2. **Syscall** là cổng vào hợp lệ từ user mode sang kernel mode
3. **Ring thôi chưa đủ**, còn phải có **page table + U/S bit + permission bits** để bảo vệ bộ nhớ

---

## 28. Một câu chốt dễ nhớ

> **User mode không trực tiếp điều khiển tài nguyên đặc quyền; nó chỉ có thể yêu cầu kernel thực hiện thay thông qua system call, và CPU + page permission sẽ cùng nhau cưỡng chế ranh giới đó.**

## 29. Từ khóa nên học tiếp nếu muốn đào sâu

Sau bài này, nếu muốn học tiếp theo hướng kernel/embedded/debug, bạn nên đi tiếp các chủ đề:

* syscall ABI trên x86-64
* stack frame khi vào kernel
* interrupt / exception / trap
* page table nhiều cấp
* TLB
* `copy_to_user()` / `copy_from_user()`
* VFS và `file_operations`
* char device / block device / network driver
* process context vs interrupt context
* user stack vs kernel stack

Các chủ đề này nối trực tiếp với protection rings và giúp bạn hiểu sâu cách Linux thật sự hoạt động.
