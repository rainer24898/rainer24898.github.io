# 1. IPC là gì và vì sao quan trọng?

**IPC (Inter-Process Communication)** là tập hợp các cơ chế cho phép các process giao tiếp, trao đổi dữ liệu và đồng bộ hoạt động với nhau.

Trong Linux, mỗi process có:

* **virtual address space riêng**
* **stack riêng**
* **heap riêng**
* **register context riêng**
* **trạng thái scheduling riêng**

Điều đó có nghĩa là:

* process A **không thể tự ý đọc/ghi** biến của process B
* muốn phối hợp công việc, phải đi qua một cơ chế do kernel hỗ trợ

Đó chính là nơi IPC xuất hiện.

# 2. Bản đồ tư duy IPC trong Linux

Có thể nhìn IPC thành 3 nhóm lớn:

```text
                         +----------------------+
                         |         IPC          |
                         +----------+-----------+
                                    |
        +---------------------------+----------------------------+
        |                           |                            |
        v                           v                            v
+------------------+      +-------------------+      +----------------------+
| Data transfer    |      | Shared state      |      | Event / Synchronize  |
+------------------+      +-------------------+      +----------------------+
| pipe             |      | shared memory     |      | semaphore            |
| FIFO             |      | mmap shared       |      | mutex/condvar        |
| message queue    |      | ring buffer       |      | signal               |
| socket           |      | lock-free queues  |      | eventfd / futex      |
+------------------+      +-------------------+      +----------------------+
```

Nếu nhìn theo mức độ điển hình trong thực tế:

* **Pipe / FIFO**: đơn giản, classic, local
* **Unix domain socket**: cực kỳ thực dụng cho service nội bộ
* **TCP socket**: khi cần mở rộng ra mạng
* **Shared memory**: throughput và latency tốt nhất nếu làm đúng
* **Semaphore / mutex / condvar**: gần như luôn cần khi có state dùng chung

# 3. Shared memory: nhanh vì sao, khó vì sao?

## 3.1 Tại sao shared memory nhanh?

Với pipe/socket/message queue, dữ liệu thường đi như sau:

```text
Process A user buffer
      |
      v
copy_from_user()
      |
      v
kernel buffer / kernel object
      |
      v
copy_to_user()
      |
      v
Process B user buffer
```

Tức là có ít nhất một lần dữ liệu phải đi qua kernel buffer, thường là hai chiều copy.

Với **shared memory**, cả hai process cùng được map vào cùng một tập physical pages:

```text
Process A VA 0x7f....  ----+
                           |----> same physical pages in RAM
Process B VA 0x7a....  ----+
```

Kernel chỉ làm nhiệm vụ:

* tạo object shared memory
* quản lý mapping
* ánh xạ vùng nhớ đó vào address space của từng process

Sau khi map xong, dữ liệu được process đọc/ghi **trực tiếp trên vùng memory chung**.

Lợi ích:

* giảm copy dữ liệu
* giảm syscall nếu thiết kế tốt
* throughput cao
* latency thấp

## 3.2 Vì sao shared memory khó hơn pipe/socket?

Vì shared memory **chỉ cho bạn vùng nhớ chung**, chứ không tự giải quyết:

* ai được ghi trước
* ai được đọc sau
* dữ liệu đã complete chưa
* có process nào đang ghi dở không
* reader có đọc phải dữ liệu rác không
* nhiều producer / nhiều consumer sẽ tranh chấp thế nào

Nói cách khác:

> Shared memory giải quyết **bài toán chia sẻ dữ liệu**, nhưng không tự giải quyết **bài toán đồng bộ truy cập**.

# 4. Semaphore: công cụ đồng bộ đi kèm shared memory

## 4.1 Semaphore là gì?

Semaphore là một counter do kernel quản lý, dùng để điều phối quyền truy cập tài nguyên.

Hai loại tư duy phổ biến:

* **binary semaphore**: giống khóa nhị phân
* **counting semaphore**: số lượng tài nguyên còn trống / còn sẵn sàng

## 4.2 Trong producer-consumer, semaphore dùng để làm gì?

Bài toán queue giới hạn kích thước thường dùng 3 semaphore:

* `empty`: số slot trống
* `full`: số slot có dữ liệu
* `mutex`: đảm bảo chỉ một process sửa cấu trúc queue tại một thời điểm

Sơ đồ:

```text
                 +----------------------------+
                 |   Shared ring buffer       |
                 |                            |
Producer ----->  | head / tail / count / data |  -----> Consumer
                 +----------------------------+
                       ^         ^        ^
                       |         |        |
                    empty       full     mutex
```

Ý nghĩa:

* Producer muốn ghi thì phải chờ `empty > 0`
* Consumer muốn đọc thì phải chờ `full > 0`
* Khi cập nhật `head/tail/count`, cần giữ `mutex`

# 5. Producer-consumer: bài toán kinh điển

Đây là mô hình xuất hiện ở khắp nơi:

* packet RX/TX pipeline
* logging queue
* telemetry
* audio/video buffering
* worker queue
* command dispatcher

## 5.1 Logic chuẩn

### Producer

1. chờ còn slot trống (`sem_wait(empty)`)
2. vào vùng critical section (`sem_wait(mutex)`)
3. ghi message vào buffer tại `tail`
4. tăng `tail`
5. tăng `count`
6. rời critical section (`sem_post(mutex)`)
7. báo có dữ liệu (`sem_post(full)`)

### Consumer

1. chờ có dữ liệu (`sem_wait(full)`)
2. vào vùng critical section (`sem_wait(mutex)`)
3. đọc message từ `head`
4. tăng `head`
5. giảm `count`
6. rời critical section (`sem_post(mutex)`)
7. trả lại slot trống (`sem_post(empty)`)

## 5.2 Vì sao không chỉ dùng mỗi mutex?

Nếu chỉ có mutex:

* producer vẫn phải tự kiểm tra full/empty
* consumer vẫn phải polling liên tục
* tốn CPU hoặc phải tự nghĩ thêm cơ chế wakeup

Semaphore giúp biến điều này thành **blocking synchronization chuẩn**.

# 6. Code C hoàn chỉnh: shared memory + semaphore + ring buffer

Dưới đây là một ví dụ **một file C duy nhất**, có thể chạy ở 3 mode:

* `init`: tạo shared memory và semaphore
* `producer`: ghi message
* `consumer`: đọc message
* `cleanup`: dọn tài nguyên IPC

Mô hình:

* dùng **POSIX shared memory**: `shm_open()` + `mmap()`
* dùng **POSIX named semaphore**: `sem_open()`
* queue là **ring buffer fixed-size**

---

## 6.1 File: `ipc_shm_sem_pc.c`

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <unistd.h>
#include <fcntl.h>
#include <errno.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <semaphore.h>

#define SHM_NAME        "/pc_demo_shm"
#define SEM_MUTEX_NAME  "/pc_demo_mutex"
#define SEM_EMPTY_NAME  "/pc_demo_empty"
#define SEM_FULL_NAME   "/pc_demo_full"

#define QUEUE_CAPACITY  8
#define MSG_SIZE        128

struct shm_queue {
    int head;
    int tail;
    int count;
    char messages[QUEUE_CAPACITY][MSG_SIZE];
};

static void die(const char *msg) {
    perror(msg);
    exit(EXIT_FAILURE);
}

static int open_or_create_shm(int create) {
    int flags = O_RDWR;
    if (create) {
        flags |= O_CREAT;
    }

    int fd = shm_open(SHM_NAME, flags, 0666);
    if (fd == -1) {
        die("shm_open");
    }

    if (create) {
        if (ftruncate(fd, sizeof(struct shm_queue)) == -1) {
            close(fd);
            die("ftruncate");
        }
    }

    return fd;
}

static struct shm_queue *map_queue(int fd) {
    void *addr = mmap(NULL,
                      sizeof(struct shm_queue),
                      PROT_READ | PROT_WRITE,
                      MAP_SHARED,
                      fd,
                      0);
    if (addr == MAP_FAILED) {
        close(fd);
        die("mmap");
    }
    return (struct shm_queue *)addr;
}

static sem_t *open_named_sem(const char *name, int create, unsigned int initval) {
    sem_t *sem;
    if (create) {
        sem = sem_open(name, O_CREAT, 0666, initval);
    } else {
        sem = sem_open(name, 0);
    }

    if (sem == SEM_FAILED) {
        die("sem_open");
    }
    return sem;
}

static void close_resources(struct shm_queue *q, int fd,
                            sem_t *mutex_sem, sem_t *empty_sem, sem_t *full_sem) {
    if (q != NULL) {
        munmap(q, sizeof(struct shm_queue));
    }
    if (fd >= 0) {
        close(fd);
    }
    if (mutex_sem != NULL) {
        sem_close(mutex_sem);
    }
    if (empty_sem != NULL) {
        sem_close(empty_sem);
    }
    if (full_sem != NULL) {
        sem_close(full_sem);
    }
}

static void do_init(void) {
    int fd = open_or_create_shm(1);
    struct shm_queue *q = map_queue(fd);

    memset(q, 0, sizeof(*q));
    q->head = 0;
    q->tail = 0;
    q->count = 0;

    sem_t *mutex_sem = open_named_sem(SEM_MUTEX_NAME, 1, 1);
    sem_t *empty_sem = open_named_sem(SEM_EMPTY_NAME, 1, QUEUE_CAPACITY);
    sem_t *full_sem  = open_named_sem(SEM_FULL_NAME, 1, 0);

    printf("[init] shared memory and semaphores created
");
    printf("[init] queue capacity = %d
", QUEUE_CAPACITY);

    close_resources(q, fd, mutex_sem, empty_sem, full_sem);
}

static void do_producer(const char *base_msg, int n_messages, int delay_ms) {
    int fd = open_or_create_shm(0);
    struct shm_queue *q = map_queue(fd);

    sem_t *mutex_sem = open_named_sem(SEM_MUTEX_NAME, 0, 0);
    sem_t *empty_sem = open_named_sem(SEM_EMPTY_NAME, 0, 0);
    sem_t *full_sem  = open_named_sem(SEM_FULL_NAME, 0, 0);

    for (int i = 0; i < n_messages; i++) {
        char msg[MSG_SIZE];
        snprintf(msg, sizeof(msg), "%s #%d (pid=%ld)", base_msg, i + 1, (long)getpid());

        if (sem_wait(empty_sem) == -1) {
            die("sem_wait(empty)");
        }
        if (sem_wait(mutex_sem) == -1) {
            die("sem_wait(mutex)");
        }

        strncpy(q->messages[q->tail], msg, MSG_SIZE - 1);
        q->messages[q->tail][MSG_SIZE - 1] = '�';

        printf("[producer %ld] write slot=%d msg=\"%s\"
",
               (long)getpid(), q->tail, q->messages[q->tail]);

        q->tail = (q->tail + 1) % QUEUE_CAPACITY;
        q->count++;

        if (sem_post(mutex_sem) == -1) {
            die("sem_post(mutex)");
        }
        if (sem_post(full_sem) == -1) {
            die("sem_post(full)");
        }

        if (delay_ms > 0) {
            usleep((useconds_t)delay_ms * 1000);
        }
    }

    close_resources(q, fd, mutex_sem, empty_sem, full_sem);
}

static void do_consumer(int n_messages, int delay_ms) {
    int fd = open_or_create_shm(0);
    struct shm_queue *q = map_queue(fd);

    sem_t *mutex_sem = open_named_sem(SEM_MUTEX_NAME, 0, 0);
    sem_t *empty_sem = open_named_sem(SEM_EMPTY_NAME, 0, 0);
    sem_t *full_sem  = open_named_sem(SEM_FULL_NAME, 0, 0);

    for (int i = 0; i < n_messages; i++) {
        char msg[MSG_SIZE];

        if (sem_wait(full_sem) == -1) {
            die("sem_wait(full)");
        }
        if (sem_wait(mutex_sem) == -1) {
            die("sem_wait(mutex)");
        }

        strncpy(msg, q->messages[q->head], sizeof(msg) - 1);
        msg[sizeof(msg) - 1] = '�';

        printf("[consumer %ld] read  slot=%d msg=\"%s\"
",
               (long)getpid(), q->head, msg);

        q->messages[q->head][0] = '�';
        q->head = (q->head + 1) % QUEUE_CAPACITY;
        q->count--;

        if (sem_post(mutex_sem) == -1) {
            die("sem_post(mutex)");
        }
        if (sem_post(empty_sem) == -1) {
            die("sem_post(empty)");
        }

        if (delay_ms > 0) {
            usleep((useconds_t)delay_ms * 1000);
        }
    }

    close_resources(q, fd, mutex_sem, empty_sem, full_sem);
}

static void do_cleanup(void) {
    shm_unlink(SHM_NAME);
    sem_unlink(SEM_MUTEX_NAME);
    sem_unlink(SEM_EMPTY_NAME);
    sem_unlink(SEM_FULL_NAME);
    printf("[cleanup] IPC resources removed
");
}

static void usage(const char *prog) {
    fprintf(stderr,
            "Usage:
"
            "  %s init
"
            "  %s producer <base_msg> <count> [delay_ms]
"
            "  %s consumer <count> [delay_ms]
"
            "  %s cleanup
",
            prog, prog, prog, prog);
    exit(EXIT_FAILURE);
}

int main(int argc, char *argv[]) {
    if (argc < 2) {
        usage(argv[0]);
    }

    if (strcmp(argv[1], "init") == 0) {
        do_init();
    } else if (strcmp(argv[1], "producer") == 0) {
        if (argc < 4) {
            usage(argv[0]);
        }
        const char *base_msg = argv[2];
        int count = atoi(argv[3]);
        int delay_ms = (argc >= 5) ? atoi(argv[4]) : 0;
        do_producer(base_msg, count, delay_ms);
    } else if (strcmp(argv[1], "consumer") == 0) {
        if (argc < 3) {
            usage(argv[0]);
        }
        int count = atoi(argv[2]);
        int delay_ms = (argc >= 4) ? atoi(argv[3]) : 0;
        do_consumer(count, delay_ms);
    } else if (strcmp(argv[1], "cleanup") == 0) {
        do_cleanup();
    } else {
        usage(argv[0]);
    }

    return 0;
}
```

# 7. Phân tích luồng chạy của ví dụ producer-consumer

## 7.1 Pha khởi tạo

`init` làm các việc:

1. `shm_open()` tạo shared memory object
2. `ftruncate()` đặt kích thước object
3. `mmap()` map object vào process init
4. `sem_open()` tạo 3 named semaphores

   * `mutex = 1`
   * `empty = QUEUE_CAPACITY`
   * `full = 0`
5. memset queue về trạng thái rỗng

## 7.2 Khi producer ghi dữ liệu

Trình tự:

```text
producer
   |
   | sem_wait(empty)
   |   nếu queue đầy -> block trong kernel
   |
   | sem_wait(mutex)
   |   vào critical section
   |
   | ghi vào q->messages[tail]
   | cập nhật tail, count
   |
   | sem_post(mutex)
   | sem_post(full)
   |   đánh thức consumer nếu đang chờ dữ liệu
   v
 return to user
```

## 7.3 Khi consumer đọc dữ liệu

```text
consumer
   |
   | sem_wait(full)
   |   nếu queue rỗng -> block trong kernel
   |
   | sem_wait(mutex)
   |   vào critical section
   |
   | đọc q->messages[head]
   | cập nhật head, count
   |
   | sem_post(mutex)
   | sem_post(empty)
   |   trả lại slot trống cho producer
   v
 return to user
```

## 7.4 Tại sao `count` không đủ để đồng bộ?

Vì `count` chỉ là trạng thái trong memory chung. Nếu không có semaphore:

* hai producer có thể cùng nhìn thấy `count < capacity`
* cả hai cùng ghi vào cùng vùng tail
* queue hỏng ngay lập tức

Nên `count` chỉ là thông tin, không phải cơ chế đồng bộ.

# 8. Những lỗi thường gặp với shared memory

## 8.1 Quên đồng bộ hóa truy cập

Ví dụ lỗi kinh điển:

* writer đang ghi 128 byte message
* reader đã thấy flag “message ready” trước khi dữ liệu hoàn chỉnh
* reader đọc phải dữ liệu nửa cũ nửa mới

## 8.2 Sai thứ tự cập nhật metadata

Ví dụ:

* tăng `tail` trước khi copy data xong
* consumer chạy vào đọc slot đó ngay

## 8.3 Không cleanup tài nguyên

POSIX shared memory và named semaphores có thể còn tồn tại sau khi process chết.

Phải nhớ:

* `shm_unlink()`
* `sem_unlink()`

## 8.4 Nhiều producer / nhiều consumer nhưng chỉ nghĩ theo 1-1

Khi mở rộng sang N producer / M consumer, cần kiểm tra lại:

* critical section đủ chặt chưa
* queue metadata có nhất quán không
* log/debug có phân biệt pid/thread không

## 8.5 Đọc/ghi struct phức tạp mà không tính alignment / cache line / false sharing

Trong hệ thống hiệu năng cao, metadata như `head/tail` đôi khi còn cần:

* tách cache line
* dùng atomic
* memory barrier rõ ràng

# 9. Pipe: byte stream qua kernel buffer

## 9.1 Pipe là gì?

`pipe()` tạo ra một cặp file descriptor:

* `fd[0]`: read end
* `fd[1]`: write end

Dữ liệu ghi vào `fd[1]` sẽ đi vào **kernel pipe buffer**, sau đó process khác đọc từ `fd[0]`.

ASCII:

```text
write(fd[1])
    |
    v
+---------------------------+
|   kernel pipe buffer      |
|   circular buffer logic   |
+---------------------------+
    |
    v
read(fd[0])
```

## 9.2 Bản chất dữ liệu của pipe

Pipe là **byte stream**, không giữ ranh giới message theo kiểu “gói tin”.

Ví dụ:

* writer 1 ghi 20 byte
* writer 2 ghi 30 byte
* reader có thể `read()` 10, rồi 25, rồi 15 byte

Do đó:

> Với pipe, nếu muốn message boundary rõ ràng, ứng dụng phải tự định nghĩa protocol.

## 9.3 Pipe dưới góc nhìn kernel

Khi gọi `pipe()`:

* kernel tạo pipe object nội bộ
* tạo 2 file object tương ứng read end / write end
* gắn vào file descriptor table của process

Sau `fork()`, child thừa hưởng các FD đó, nên pipe cực hợp cho parent-child.

ASCII:

```text
parent process
  fd table
   3 --> read end  ----+
   4 --> write end ----+--> kernel pipe object/buffer

fork()

child process
  fd table
   3 --> read end  ----+
   4 --> write end ----+
```

## 9.4 `read()` và `write()` với pipe

### `write(pipe_write_fd, buf, n)`

Kernel sẽ:

1. kiểm tra pipe còn reader không
2. nếu không còn reader, gửi `SIGPIPE` hoặc trả `EPIPE`
3. nếu buffer còn chỗ, copy dữ liệu vào pipe buffer
4. nếu buffer đầy:

   * blocking mode: sleep chờ chỗ trống
   * non-blocking: trả `EAGAIN`
5. nếu có reader đang ngủ, đánh thức reader

### `read(pipe_read_fd, buf, n)`

Kernel sẽ:

1. nếu pipe buffer có dữ liệu, copy ra user buffer
2. nếu buffer rỗng nhưng còn writer:

   * blocking: sleep chờ dữ liệu
   * non-blocking: trả `EAGAIN`
3. nếu không còn writer và buffer rỗng: trả `0` (EOF)

## 9.5 Vì sao hay bị treo khi dùng pipe sai?

Lỗi rất phổ biến:

* parent đóng thiếu đầu pipe
* child đóng thiếu đầu pipe
* reader chờ EOF mãi không thấy vì vẫn còn một write end mở đâu đó

Đây là bug cực hay gặp khi làm `fork()` + `pipe()`.

# 10. FIFO: named pipe trong filesystem

## 10.1 FIFO là gì?

FIFO là pipe có tên trong filesystem, tạo bằng `mkfifo()`.

Ví dụ:

```bash
mkfifo /tmp/demo_fifo
```

Nó cho phép hai process **không cần quan hệ cha-con** vẫn mở cùng một object IPC.

## 10.2 Bản chất kernel của FIFO

Dù xuất hiện như một file path, FIFO không phải file thường chứa dữ liệu trên disk.

Nó là:

* một inode kiểu special file
* dữ liệu thực tế vẫn đi qua kernel pipe-style buffer
* read/write semantics gần với pipe

ASCII:

```text
Process A open("/tmp/demo_fifo", O_WRONLY)
        |
        +------> VFS path lookup -----> FIFO inode -----> kernel FIFO/pipe logic
        |
Process B open("/tmp/demo_fifo", O_RDONLY)
```

## 10.3 Điểm khác biệt lớn giữa pipe và FIFO

* **pipe**: thường tạo tạm thời, sống theo FD / process
* **FIFO**: có tên trong filesystem, process độc lập có thể kết nối vào sau

Nhưng về dữ liệu:

* cả hai đều là **byte stream**
* đều dựa vào kernel buffer kiểu pipe

# 11. Socket: endpoint giao tiếp tổng quát nhất

Socket là cơ chế giao tiếp tổng quát hơn pipe/FIFO.

Có hai nhánh chính rất quan trọng:

* **Unix domain socket**: giao tiếp nội bộ trong cùng host
* **TCP/UDP socket**: giao tiếp qua mạng

## 11.1 Unix domain socket

Đặc điểm:

* local only
* full-duplex
* hỗ trợ mô hình client/server
* có thể truyền file descriptor (`SCM_RIGHTS`)
* rất hợp cho daemon/service nội bộ

## 11.2 TCP socket

Đặc điểm:

* stream socket
* dữ liệu đi qua network stack
* có send buffer / receive buffer trong kernel
* có congestion control, retransmission, flow control

## 11.3 UDP socket

Đặc điểm:

* datagram
* giữ ranh giới gói tin
* không bảo đảm delivery như TCP

## 11.4 Vì sao socket mạnh hơn pipe?

Vì socket hỗ trợ:

* client/server nhiều endpoint
* full-duplex tự nhiên
* local hoặc remote
* multiplexing tốt với `select/poll/epoll`
* control message, credential passing, fd passing

# 12. Góc nhìn syscall → kernel object → wakeup

Đây là phần cốt lõi nếu muốn hiểu IPC ở mức kernel-facing.

## 12.1 Luồng chung của một syscall IPC

Dù bạn dùng pipe, FIFO, socket hay semaphore, mô hình tổng quát thường là:

```text
user code
   |
   v
libc wrapper (nếu có)
   |
   v
syscall entry
   |
   v
kernel IPC subsystem
   |
   +--> validate fd / object
   +--> check permissions/state
   +--> copy_from_user / copy_to_user (nếu cần)
   +--> enqueue/dequeue data
   +--> sleep or wake up tasks
   |
   v
return to user space
```

## 12.2 Khi `write()` vào pipe/socket

Có vài bước tư duy luôn lặp lại:

1. kernel xác định FD đang trỏ tới object nào
2. kiểm tra object có cho phép ghi không
3. kiểm tra buffer kernel còn chỗ không
4. nếu còn chỗ: copy dữ liệu từ user vào kernel buffer
5. nếu đầy:

   * non-blocking: trả `EAGAIN`
   * blocking: đưa task vào wait queue và sleep
6. đánh thức reader nếu reader đang ngủ

## 12.3 Khi `read()` từ pipe/socket

1. kernel xác định read end / socket receive queue
2. nếu có dữ liệu: copy ra user buffer
3. nếu chưa có dữ liệu:

   * non-blocking: `EAGAIN`
   * blocking: đưa task vào wait queue và sleep
4. khi writer/network stack đẩy dữ liệu tới, kernel wakeup task đó

## 12.4 “Blocking” thật ra là gì?

Ở mức kernel, blocking thường nghĩa là:

* task không còn tiếp tục chạy user code tại thời điểm đó
* task được chuyển sang trạng thái ngủ phù hợp
* scheduler chọn task khác để chạy
* khi sự kiện xảy ra, task được đánh thức và quay lại runnable state

ASCII:

```text
Task A: read(pipe)
   |
   +--> no data
   +--> sleep on wait queue

Task B: write(pipe)
   |
   +--> data inserted into kernel buffer
   +--> wake up Task A

Scheduler
   |
   +--> Task A runnable again
```

## 12.5 Chi phí thực tế của IPC không chỉ là copy byte

Một thao tác IPC có thể tốn:

* syscall overhead
* copy_from_user / copy_to_user
* cache miss
* lock contention trong kernel object
* wakeup cost
* context switch
* scheduler latency

Đó là lý do vì sao shared memory thường thắng khi dữ liệu lớn và tần suất cao.

# 13. So sánh pipe / FIFO / Unix socket / TCP socket

## 13.1 Pipe

* local only
* rất hợp cho parent-child
* unidirectional theo mô hình đơn giản
* byte stream
* không có tên trong filesystem
* kernel buffer tương đối gọn

## 13.2 FIFO

* local only
* process độc lập có thể kết nối qua path
* byte stream
* phù hợp bài toán đơn giản
* semantics open/read/write có chút phiền hơn pipe thường

## 13.3 Unix domain socket

* local only
* full-duplex
* client/server tốt
* hỗ trợ `socketpair()`
* hỗ trợ `sendmsg()/recvmsg()` nâng cao
* rất hợp cho daemon/control plane nội bộ

## 13.4 TCP socket

* local hoặc remote
* full-duplex
* stream
* qua network stack nên phức tạp hơn
* mở rộng hệ thống tốt nhất

# 14. Lời khuyên chọn IPC trong hệ thống thực tế

## 14.1 Khi nào chọn shared memory?

Chọn khi:

* dữ liệu lớn
* tần suất trao đổi cao
* cần throughput/latency tốt
* chấp nhận complexity cao hơn

Ví dụ:

* packet processing
* media pipeline
* shared telemetry buffer
* high-frequency producer-consumer

## 14.2 Khi nào chọn pipe/FIFO?

Chọn khi:

* giao tiếp đơn giản
* topology không phức tạp
* cần làm nhanh, rõ ràng
* parent-child hoặc 2 process local đơn giản

## 14.3 Khi nào chọn Unix domain socket?

Đây là lựa chọn rất cân bằng khi:

* cần local IPC mạnh và linh hoạt
* cần request/response
* muốn kiến trúc service/daemon
* có thể mở rộng logic event loop / epoll

## 14.4 Khi nào chọn TCP socket?

Chọn khi:

* có khả năng scale ra nhiều machine
* cần client/server chuẩn mạng
* cần giao thức mở rộng được

# 15. Kết luận

Nếu phải gói toàn bộ bài này trong vài ý cốt lõi, thì có thể nhớ như sau:

## Ý 1: Shared memory nhanh vì bỏ qua đường copy dữ liệu qua kernel buffer

Nhưng nó không miễn phí về mặt tư duy. Bạn phải tự giải quyết:

* synchronization
* message ownership
* ordering
* metadata consistency

## Ý 2: Semaphore là mảnh ghép rất tự nhiên cho producer-consumer

Bộ ba kinh điển:

* `empty`
* `full`
* `mutex`

chính là nền tảng của vô số queue thực tế.

## Ý 3: Pipe / FIFO / socket đều có điểm gặp nhau ở kernel buffer và wait queue

Sự khác nhau nằm ở:

* cách tạo object
* topology kết nối
* stream hay datagram
* local hay remote
* độ linh hoạt của API

## Ý 4: Chọn IPC không phải chọn API, mà là chọn trade-off

Bạn đang trade-off giữa:

* đơn giản vs hiệu năng
* local vs remote
* stream vs message
* ít code vs khó debug
* latency vs độ linh hoạt

# Phụ lục B – Hướng đào sâu tiếp theo

Nếu muốn mở rộng tiếp bài này theo hướng kernel-level hơn nữa, rất nên viết thêm các phần sau:

1. **`fork()` / `exec()` / `waitpid()` và vòng đời file descriptor**
2. **`dup()` / `dup2()` / redirection / shell pipeline**
3. **`socketpair()` và Unix domain socket nâng cao**
4. **`epoll` + `eventfd` + non-blocking I/O**
5. **`futex` và cách pthread mutex map xuống kernel**
6. **memory ordering / atomic / cache coherence trong shared memory lock-free**


> IPC không chỉ là danh sách syscall. Nó là cách kernel cho các thực thể cô lập phối hợp với nhau một cách có kiểm soát, hiệu quả, và có thể debug được.
