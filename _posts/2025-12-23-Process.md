## 1. Mở đầu: program không phải process

Rất nhiều tài liệu nhập môn nói rằng:

* **Program** là chương trình
* **Process** là chương trình đang chạy

Câu này đúng, nhưng còn quá nông.

Trong Linux, một **process** không chỉ là mã lệnh đang chạy, mà là một **thực thể được kernel quản lý** với đầy đủ:

* PID
* virtual address space
* page table
* file descriptor table
* signal state
* scheduling state
* kernel stack
* CPU context
* credentials
* quan hệ parent/child/sibling

Nói cách khác:

> File ELF nằm trên đĩa chỉ là input. Khi kernel nạp và quản lý nó, thực thể sống thật sự trong hệ thống là **task/process**.

## 2. Process là gì dưới góc nhìn kernel

### 2.1. Góc nhìn logic

Một process thường được nhìn theo hai lớp:

* **User view**: chương trình đang chạy
* **Kernel view**: một task có state, resource và execution context riêng

### 2.2. Những gì một process thường có

Ở mức user-space, ta hay nghĩ tới:

* text segment
* data segment
* bss
* heap
* stack

Nhưng ở mức kernel, còn nhiều thứ quan trọng hơn:

* `task_struct`
* `mm_struct` cho không gian nhớ
* `files_struct` cho file descriptor table
* `fs_struct` cho cwd/root
* `signal_struct` và `sighand_struct`
* `sched_entity` hoặc cấu trúc scheduling tương ứng
* kernel stack
* saved register context

### 2.3. ASCII nhìn nhanh

```text
+----------------------------------------------------------+
|                    User Process View                     |
+----------------------------------------------------------+
| text | rodata | data | bss | heap ---> ... <--- stack    |
+----------------------------------------------------------+
            |                 syscall / exception / signal
            v
+----------------------------------------------------------+
|                   Kernel Process View                    |
+----------------------------------------------------------+
| task_struct                                              |
|   |- pid / tgid                                          |
|   |- state                                               |
|   |- mm ----------> mm_struct ----------> page tables    |
|   |- files -------> files_struct -------> fd table       |
|   |- signal ------> signal_struct                        |
|   |- parent / children / sibling                         |
|   |- se / rt / dl scheduling info                        |
|   |- stack -------> kernel stack                         |
|   |- thread ------> saved CPU context                    |
+----------------------------------------------------------+
```

## 3. User space, kernel space và syscall path

Linux tách hệ thống thành hai miền đặc quyền rõ ràng:

### 3.1. User space

User space là nơi ứng dụng chạy.

Đặc điểm:

* privilege thấp
* không truy cập trực tiếp hardware theo ý muốn
* không đọc/ghi kernel memory trực tiếp
* muốn dùng dịch vụ hệ thống phải đi qua syscall

### 3.2. Kernel space

Kernel space là nơi kernel và kernel module chạy.

Tại đây kernel có toàn quyền:

* quản lý CPU
* lập lịch task
* cấp phát/tracking memory
* quản lý file system
* quản lý driver và interrupt
* tạo/kết thúc task
* thiết lập page table và protection

### 3.3. Luồng syscall

Ví dụ một syscall kiểu `read()`:

```text
user code
  -> libc wrapper
    -> syscall instruction
      -> CPU chuyển sang kernel mode
        -> syscall entry
          -> sys_read / ksys_read / vfs_read / driver
        -> return value set vào register
      -> CPU quay lại user mode
```

### 3.4. Điểm rất hay bị hiểu nhầm

**Kernel code là dùng chung cho toàn hệ thống**, nhưng **kernel context là riêng cho từng task**.

Ví dụ 100 process cùng gọi `read()`:

* tất cả đều đi qua cùng một code path trong kernel
* nhưng kernel vẫn biết task nào đang chạy nhờ:

  * `current`
  * `task_struct`
  * kernel stack hiện tại
  * page table hiện tại
  * file descriptor table tương ứng

ASCII nhìn theo góc này:

```text
Process A ---- syscall(read) ----+
                                 |
Process B ---- syscall(read) ----+----> shared kernel code path
                                 |
Process C ---- syscall(read) ----+
                                          |
                                          v
                                 current -> task_struct của task hiện tại
```

## 4. `fork()` thực sự làm gì

### 4.1. Ý nghĩa user-space

`fork()` tạo ra một child process từ parent process.

Giá trị trả về:

* `-1`: lỗi
* `0`: đang ở child
* `> 0`: đang ở parent, giá trị là PID của child

### 4.2. Ví dụ C cơ bản

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>

int main(void)
{
    pid_t pid = fork();

    if (pid < 0)
    {
        perror("fork");
        return 1;
    }
    else if (pid == 0)
    {
        printf("[child] pid=%d ppid=%d
", getpid(), getppid());
    }
    else
    {
        printf("[parent] pid=%d child=%d
", getpid(), pid);
    }

    return 0;
}
```

### 4.3. Góc nhìn kernel-level

Khi `fork()` xảy ra, Linux không làm kiểu “copy nguyên xi toàn bộ process ngay lập tức”. Thay vào đó, kernel tạo ra một task mới với nhiều phần được:

* copy
* reference
* share tạm thời
* lazy-copy bằng Copy-on-Write

Các bước khái quát:

1. allocate task mới
2. tạo `task_struct` mới
3. cấp PID/TID
4. chuẩn bị kernel stack cho child
5. copy register context cần thiết
6. copy/share `mm`, `files`, `fs`, `signal` tùy semantics
7. set return value khác nhau cho cha và con
8. đưa child vào run queue

### 4.4. ASCII: điều gì được “nhân bản” sau `fork()`

```text
Before fork:

 Parent
   |
   +-- task_struct
   +-- mm_struct ----> page tables ----> physical pages
   +-- files_struct -> fd table
   +-- kernel stack

After fork (conceptual):

 Parent                         Child
   |                              |
   +-- task_struct                +-- task_struct (new)
   +-- mm_struct -----+           +-- mm_struct (copied metadata)
   +-- files_struct --+----+      +-- files_struct (ref/copied)
   +-- kernel stack        |      +-- kernel stack (new)
                           |
                           +--> user pages initially shared (COW)
```

### 4.5. Copy-on-Write (COW)

Sau `fork()`, parent và child không bị copy toàn bộ RAM ngay.

Thay vào đó:

* page table của child được thiết lập để trỏ vào cùng các page vật lý
* các page được đánh dấu read-only phù hợp
* khi một bên ghi, page fault xảy ra
* kernel tạo page copy thật cho bên ghi

Lợi ích:

* giảm chi phí `fork()`
* tiết kiệm RAM
* cực kỳ hiệu quả cho pattern `fork() -> exec()`

## 5. `exec()` thay image của process ra sao

Sau `fork()`, child thường không giữ nguyên image cũ mà gọi `exec*()` để thay bằng chương trình mới.

Các hàm họ `exec()` chỉ là wrapper; lõi thực sự là `execve()`.

### 5.1. Điều gì thay đổi sau `exec()`

Sau khi `exec()` thành công:

* code cũ mất
* data cũ mất
* heap/stack cũ mất
* memory map cũ bị thay mới
* signal handler phần lớn reset theo rule của POSIX/Linux
* PID **không đổi**
* process identity theo nghĩa “task đang sống” vẫn còn, nhưng image đã thay

### 5.2. ASCII: `fork()` rồi `exec()`

```text
Parent process
    |
    +-- fork() ---> Child process (ban đầu giống parent)
                        |
                        +-- execve("/bin/ls", ...)
                                |
                                v
                       Child vẫn cùng PID đó,
                       nhưng address space đã là image mới
```

### 5.3. Ví dụ C nhỏ

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>

int main(void)
{
    pid_t pid = fork();

    if (pid < 0)
    {
        perror("fork");
        return 1;
    }
    else if (pid == 0)
    {
        char *argv[] = {"ls", "-l", NULL};
        execvp("ls", argv);
        perror("execvp");
        _exit(127);
    }
    else
    {
        int status = 0;
        waitpid(pid, &status, 0);
    }

    return 0;
}
```

## 6. `exit()`, `_exit()` và vì sao zombie xuất hiện

### 6.1. `exit()`

`exit()` là hàm của libc. Nó thường:

* flush stdio buffer
* gọi callback trong `atexit()`
* cleanup theo runtime C
* cuối cùng mới kết thúc process ở mức kernel

### 6.2. `_exit()`

`_exit()` đi thẳng xuống kernel để kết thúc process, không làm cleanup của libc.

Trong child sau `fork()`, nếu `exec()` fail thì thường dùng `_exit()` thay vì `exit()` để tránh:

* flush buffer lặp
* in log lặp
* phá trạng thái I/O thừa hưởng từ parent

### 6.3. Zombie sinh ra thế nào

Khi child kết thúc, kernel **không xóa ngay mọi dấu vết của child** nếu parent còn chưa lấy exit status.

Lúc đó child vào trạng thái zombie:

* không chạy nữa
* không tiêu CPU
* nhưng vẫn chiếm PID và entry trong process table
* giữ exit status và một số accounting info

ASCII:

```text
child running
   |
   +-- exit/_exit
          |
          v
   child becomes ZOMBIE
          |
          +-- parent calls wait/waitpid --> child reaped --> entry removed
```

## 7. `wait()` / `waitpid()` và hành vi reap child

### 7.1. Bản chất

Parent không chỉ “chờ” child; chính xác hơn là parent **reap** child đã chết.

### 7.2. `wait()`

* chờ một child bất kỳ
* blocking mặc định

### 7.3. `waitpid()`

Linh hoạt hơn:

* chờ đúng PID
* có thể dùng `WNOHANG`
* có thể kết hợp với các trạng thái khác

### 7.4. Một số mode quan trọng

* `waitpid(pid, &status, 0)` : chờ đúng `pid`
* `waitpid(-1, &status, 0)` : chờ bất kỳ child nào
* `waitpid(-1, &status, WNOHANG)` : không block

### 7.5. Macro đọc `status`

* `WIFEXITED(status)`
* `WEXITSTATUS(status)`
* `WIFSIGNALED(status)`
* `WTERMSIG(status)`
* `WIFSTOPPED(status)`
* `WSTOPSIG(status)`

## 8. `SIGCHLD` và mô hình reap không blocking

Khi child thay đổi trạng thái quan trọng, parent thường nhận `SIGCHLD`.

Pattern điển hình của server/process manager:

* cài handler cho `SIGCHLD`
* trong handler hoặc loop chính, gọi `waitpid(-1, &status, WNOHANG)` lặp nhiều lần
* thu dọn tất cả child đã chết

Lý do phải lặp nhiều lần:

* một lần `SIGCHLD` có thể tương ứng với nhiều child đã chết gần nhau

ASCII:

```text
child #1 exit ---+
                  |
child #2 exit ---+----> parent nhận SIGCHLD
                  |
child #3 exit ---+
                         |
                         v
            while (waitpid(-1, &st, WNOHANG) > 0) { reap all; }
```

## 9. Ví dụ C hoàn chỉnh: `fork + exec + waitpid + SIGCHLD`

Ví dụ dưới đây thể hiện đủ các ý:

* parent tạo nhiều child
* child gọi `exec()` chạy lệnh khác
* parent cài `SIGCHLD` handler
* parent reap child không blocking bằng `waitpid(..., WNOHANG)`
* xử lý đúng `errno`
* child dùng `_exit()` khi `exec()` lỗi

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <signal.h>
#include <errno.h>
#include <string.h>

static volatile sig_atomic_t g_children_left = 0;

static void sigchld_handler(int signo)
{
    int saved_errno = errno;
    int status = 0;
    pid_t pid;

    (void)signo;

    for (;;)
    {
        pid = waitpid(-1, &status, WNOHANG);

        if (pid > 0)
        {
            if (g_children_left > 0)
            {
                --g_children_left;
            }

            if (WIFEXITED(status))
            {
                printf("[parent] reaped child pid=%d exit_code=%d
",
                       pid, WEXITSTATUS(status));
            }
            else if (WIFSIGNALED(status))
            {
                printf("[parent] reaped child pid=%d killed_by_signal=%d
",
                       pid, WTERMSIG(status));
            }
        }
        else if (pid == 0)
        {
            break;
        }
        else
        {
            if (errno == ECHILD)
            {
                break;
            }
            break;
        }
    }

    errno = saved_errno;
}

static int install_sigchld_handler(void)
{
    struct sigaction sa;

    memset(&sa, 0, sizeof(sa));
    sa.sa_handler = sigchld_handler;
    sa.sa_flags = SA_RESTART | SA_NOCLDSTOP;
    sigemptyset(&sa.sa_mask);

    if (sigaction(SIGCHLD, &sa, NULL) < 0)
    {
        perror("sigaction(SIGCHLD)");
        return -1;
    }

    return 0;
}

static pid_t spawn_exec_child(const char *path, char *const argv[])
{
    pid_t pid = fork();

    if (pid < 0)
    {
        perror("fork");
        return -1;
    }

    if (pid == 0)
    {
        printf("[child ] pid=%d ppid=%d exec: %s
", getpid(), getppid(), path);
        execvp(path, argv);

        perror("execvp failed");
        _exit(127);
    }

    return pid;
}

int main(void)
{
    char *const argv1[] = {"/bin/sh", "-c", "echo child1 && sleep 1 && exit 3", NULL};
    char *const argv2[] = {"/bin/sh", "-c", "echo child2 && sleep 2 && exit 7", NULL};
    char *const argv3[] = {"/bin/sh", "-c", "echo child3 && sleep 3 && exit 0", NULL};

    if (install_sigchld_handler() < 0)
    {
        return 1;
    }

    if (spawn_exec_child("/bin/sh", argv1) < 0)
    {
        return 1;
    }
    ++g_children_left;

    if (spawn_exec_child("/bin/sh", argv2) < 0)
    {
        return 1;
    }
    ++g_children_left;

    if (spawn_exec_child("/bin/sh", argv3) < 0)
    {
        return 1;
    }
    ++g_children_left;

    printf("[parent] pid=%d spawned 3 children
", getpid());

    while (g_children_left > 0)
    {
        printf("[parent] doing other work, children_left=%d
", g_children_left);
        sleep(1);
    }

    printf("[parent] all children reaped, exiting
");
    return 0;
}
```

### Cách build

```bash
gcc -Wall -Wextra -O2 -o proc_demo proc_demo.c
```

### Điều cần lưu ý trong ví dụ

1. Trong handler signal, chỉ nên làm việc thật cẩn thận; `printf()` trong signal handler không phải lựa chọn tối ưu về mặt async-signal-safe trong code production. Ở đây dùng để minh họa luồng xử lý.
2. Production code thường dùng self-pipe/signalfd/event loop để xử lý signal sạch hơn.
3. `SA_NOCLDSTOP` giúp bỏ qua tín hiệu khi child chỉ bị stop/continue.
4. `SA_RESTART` giúp nhiều syscall đang block được restart.

## 10. `task_struct`: trung tâm quản lý task của Linux

Trong Linux, process và thread đều được biểu diễn bằng `task_struct`.

Đây là một trong những cấu trúc quan trọng nhất của kernel scheduler/process subsystem.

### 10.1. Nhìn khái quát

```text
task_struct
   |
   +-- pid / tgid
   +-- state
   +-- flags
   +-- prio / policy
   +-- mm / active_mm
   +-- files
   +-- fs
   +-- signal / sighand
   +-- parent / real_parent
   +-- children / sibling list
   +-- thread (arch-specific CPU state)
   +-- stack
   +-- sched entity / rt entity / dl entity
   +-- cpus_mask / affinity
   +-- cred
```

### 10.2. Một số trường quan trọng theo ý nghĩa

#### PID và TGID

* **PID**: ID của task
* **TGID**: thread-group ID, thường chính là PID của thread leader

Vì thế trong một process multi-thread:

* mỗi thread có PID riêng ở mức kernel
* nhưng cùng TGID để biểu diễn “cùng một process” ở góc nhìn user-space

#### `mm` và `active_mm`

* `mm` trỏ tới `mm_struct` của user address space
* kernel thread thường không có `mm` user-space theo nghĩa bình thường
* khi cần, kernel thread có thể mượn `active_mm`

#### `files`

Trỏ tới bảng file descriptor.

Nếu hai task share `files_struct`, chúng thấy cùng fd table.

#### `signal` / `sighand`

Quản lý signal pending, disposition, shared signal state.

#### `stack`

Mỗi task có **kernel stack riêng**. Đây là điểm rất quan trọng.

Dù toàn hệ thống dùng chung kernel code, mỗi task khi vào kernel vẫn có stack kernel riêng để giữ:

* local variable mức kernel
* call chain
* trap frame / saved registers

### 10.3. `current`

Macro `current` cho biết task hiện tại trên CPU đang chạy code kernel.

Nó là chìa khóa giúp kernel biết:

* syscall này thuộc task nào
* credentials nào đang active
* signal nào pending
* kernel stack nào đang dùng
* page table nào đang active

## 11. `clone()` và vì sao thread trong Linux thực chất cũng là task

### 11.1. Ý tưởng cốt lõi

Linux không có ranh giới “process vs thread” cứng theo kiểu hai thực thể hoàn toàn khác nhau.

Thay vào đó, Linux có một primitive mạnh hơn: **tạo task mới với mức độ chia sẻ tài nguyên do cờ quyết định**.

Primitive đó chính là `clone()`.

### 11.2. `clone()` chia sẻ những gì?

Tùy cờ truyền vào, task con có thể share:

* address space: `CLONE_VM`
* file descriptor table: `CLONE_FILES`
* filesystem context: `CLONE_FS`
* signal handlers: `CLONE_SIGHAND`
* thread group: `CLONE_THREAD`
* parent/child semantics đặc biệt khác

### 11.3. So sánh trực giác

* `fork()` ~ tạo task mới với semantics gần như process độc lập
* `pthread_create()` ~ tạo task mới bằng `clone()` với nhiều resource share mạnh

### 11.4. ASCII: `fork()` vs `clone()`

```text
fork():
  parent
    |
    +--> child process
          |- task_struct mới
          |- address space riêng về mặt logical
          |- fd table copy/share theo rule fork
          |- signal state phù hợp process

clone(CLONE_VM | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD):
  thread A
    |
    +--> thread B
          |- task_struct mới
          |- share cùng mm_struct
          |- share cùng fd table
          |- share signal handlers
          |- cùng thread group
```

### 11.5. Ý nghĩa thực chiến

Vì thread Linux ở mức kernel cũng là task:

* mỗi thread có scheduling entity riêng
* mỗi thread có kernel stack riêng
* mỗi thread có TID/PID riêng ở mức kernel
* scheduler schedule **thread**, không phải chỉ schedule “process” theo kiểu mơ hồ

---

## 12. Process vs thread vs kernel thread vs interrupt context

Đây là chỗ cực hay bị lẫn.

### 12.1. User process

* có user address space riêng
* có thể có một hoặc nhiều thread
* chạy user code ở user mode
* vào kernel qua syscall/interrupt/fault

### 12.2. User thread

* là thread thuộc một process
* share address space với các thread khác trong cùng process
* có kernel schedulable entity riêng
* có kernel stack riêng

### 12.3. Kernel thread

* chạy hoàn toàn trong kernel space
* không chạy user code
* thường không có user `mm` theo nghĩa thường
* dùng cho background work của kernel

### 12.4. Interrupt context

* **không phải process**
* **không phải thread bình thường**
* xảy ra khi CPU xử lý interrupt
* không có quyền sleep tùy ý như process context
* phải xử lý nhanh, phần nặng đẩy xuống bottom half / workqueue / kthread

### 12.5. Bảng so sánh

| Thực thể                | User address space       |          Kernel stack riêng |           Scheduler quản lý |        Có thể sleep | Chạy user code |
| ----------------------- | ------------------------ | --------------------------: | --------------------------: | ------------------: | -------------: |
| User process (1 thread) | có                       |                          có |                          có |                  có |             có |
| User thread             | share với process        |                          có |                          có |                  có |             có |
| Kernel thread           | không theo nghĩa user mm |                          có |                          có |                  có |          không |
| Interrupt context       | không                    | dùng stack ngữ cảnh phù hợp | không theo kiểu task thường | rất hạn chế / không |          không |

## 13. Scheduler nhìn task như thế nào

### 13.1. Scheduler không quan tâm “marketing term”

Ở mức thấp, scheduler chủ yếu quan tâm tới **task runnable nào** sẽ được chạy tiếp trên CPU.

### 13.2. Run queue

Mỗi CPU thường có run queue riêng.

ASCII đơn giản:

```text
CPU0 runqueue: [task A] [task C] [kworker/0:1] [task G]
CPU1 runqueue: [task B] [task D] [task E] [task F]
```

### 13.3. Các lớp scheduling

Linux có nhiều lớp scheduling lớn, ví dụ:

* fair class (CFS, cho task thường)
* real-time class (`SCHED_FIFO`, `SCHED_RR`)
* deadline class (`SCHED_DEADLINE`)
* idle class

### 13.4. CFS nhìn task ra sao

Với task thường, CFS cố gắng phân CPU “công bằng” bằng cách dựa trên **virtual runtime**.

Ý tưởng trực giác:

* task chạy càng nhiều thì `vruntime` càng tăng
* task có `vruntime` nhỏ hơn sẽ được ưu tiên chọn chạy tiếp

ASCII:

```text
red-black tree ordered by vruntime

      [task D: vr=10]
       /            \
 [task A:12]      [task C:18]
     /                \
[task B:14]         [task E:21]

=> task gần bên trái nhất có xu hướng được chọn trước
```

### 13.5. RT scheduler

Với `SCHED_FIFO` / `SCHED_RR`:

* ưu tiên theo priority tĩnh cao/thấp
* task realtime có thể preempt task thường
* nếu dùng sai dễ làm starve hệ thống

### 13.6. Điều quan trọng cho người làm embedded

Nếu hệ thống có:

* quá nhiều thread
* lock contention cao
* affinity đặt không hợp lý
* realtime thread priority sai

thì latency sẽ tăng rất mạnh dù CPU load nhìn ngoài có thể chưa quá cao.

## 14. Context switch ở mức kernel

### 14.1. Định nghĩa

Context switch là quá trình kernel dừng task A và khôi phục task B để CPU tiếp tục chạy B.

### 14.2. Những gì thường phải save/restore

Tùy kiến trúc CPU, kernel sẽ cần lưu/khôi phục:

* general-purpose registers
* stack pointer
* instruction pointer / PC
* flags/status register
* TLS / thread-local state
* FPU/SIMD state khi cần
* scheduler/accounting state

### 14.3. ASCII: context switch

```text
CPU currently running task A

   registers = context of A
   kernel stack = stack A
   current = A

          schedule()
              |
              v
      pick_next_task()
              |
              v
         switch_to(A, B)

After switch:

   registers = context of B
   kernel stack = stack B
   current = B
```

### 14.4. Không chỉ là save register

Chi phí context switch không chỉ nằm ở việc đổi vài thanh ghi.

Nó còn gây:

* cache pollution
* TLB disturbance
* branch predictor disturbance
* lock/scheduler overhead
* NUMA / migration penalty nếu task bị chuyển CPU

### 14.5. Context switch vs mode switch

Hai thứ này khác nhau:

#### Mode switch

Ví dụ user -> kernel do syscall:

* vẫn cùng task
* chỉ đổi privilege mode và stack/context phù hợp

#### Context switch

* đổi từ task A sang task B
* thay `current`
* thay kernel stack hiện hành
* khôi phục execution context khác

ASCII phân biệt:

```text
Mode switch:
  task A user mode -> task A kernel mode -> task A user mode

Context switch:
  task A kernel/user -> scheduler -> task B kernel/user
```

### 14.6. Vì sao multi-thread không phải lúc nào cũng nhanh hơn

Nếu sinh quá nhiều thread:

* scheduler phải quản lý nhiều runnable entity hơn
* cache locality kém đi
* lock contention tăng
* context switch tăng

Kết quả là throughput có thể giảm thay vì tăng.

## 15. Góc nhìn Embedded Linux

Embedded Linux thường có các ràng buộc:

* RAM ít
* CPU giới hạn
* yêu cầu boot time thấp
* yêu cầu độ trễ có thể rất chặt
* tài nguyên PID/process table không nên bị lãng phí

### 15.1. Hệ quả thiết kế

Trong embedded, thường cần cân nhắc rất kỹ:

* có thực sự cần thêm process mới không?
* thread có đủ không?
* có đang tạo zombie vì quên reap child không?
* có đang dùng `fork()` quá nhiều trong đường hot path không?
* có nên dùng `vfork()` cho pattern `fork()+exec()`?

### 15.2. Vì sao zombie nguy hiểm hơn trên embedded

Trên desktop, vài zombie thường chưa thấy ngay hậu quả.

Trên embedded:

* process table có thể nhỏ
* daemon tạo child định kỳ nhiều
* hệ thống chạy lâu không reboot

=> zombie leak có thể dần dần làm hệ thống không spawn được process mới.

### 15.3. Vì sao context switch đắt hơn ta tưởng

Nhiều hệ embedded không có cache/TLB lớn như server CPU.

Vì thế:

* scheduler noise
* cache miss
* migration
* lock contention

có thể đẩy latency tăng mạnh, đặc biệt với pipeline radio, multimedia, packet processing hoặc control loop thời gian thực.

## 16. Checklist debug thực chiến

Khi debug process/thread trên Linux, đặc biệt trên hệ thống nhúng, các câu hỏi nên tự kiểm tra là:

### 16.1. Nếu nghi zombie

* parent có gọi `wait()` / `waitpid()` không?
* có cài `SIGCHLD` handler đúng không?
* handler có reap hết child bằng vòng lặp không?
* có trường hợp signal đến dồn mà chỉ reap một child không?

### 16.2. Nếu nghi `fork()` lỗi

* `RLIMIT_NPROC` có chạm ngưỡng không?
* PID namespace / cgroup / policy có giới hạn không?
* memory pressure có cao không?
* page table / kernel memory có thiếu không?

### 16.3. Nếu nghi thread quá nhiều gây chậm

* số runnable thread là bao nhiêu?
* có lock contention không?
* context switch rate bao nhiêu?
* affinity có đang phá cache locality không?
* có realtime thread starve task khác không?

### 16.4. Tool hữu ích

* `ps -eLf`
* `pstree -p`
* `top`, `htop`
* `strace`
* `gdb`
* `/proc/<pid>/status`
* `/proc/<pid>/task`
* `/proc/<pid>/maps`
* `perf sched`
* `perf stat`
* `ftrace`
* `trace-cmd`

## 17. Tổng kết

Nếu phải cô đọng toàn bộ bài viết này thành vài ý cốt lõi, thì đó là:

### 17.1. Process không chỉ là chương trình đang chạy

Nó là một **task được kernel quản lý**, có memory context, scheduling context, signal state, file table, kernel stack và nhiều metadata khác.

### 17.2. `fork()` không copy thô mọi thứ ngay lập tức

Linux dùng nhiều cơ chế tối ưu, đặc biệt là **Copy-on-Write**, để tạo child hiệu quả.

### 17.3. `exec()` không tạo process mới

Nó thay image của process hiện tại. PID vẫn giữ nguyên.

### 17.4. Zombie xuất hiện vì parent chưa reap child

Muốn xóa zombie, phải gọi `wait()` / `waitpid()`.

### 17.5. Thread trong Linux về bản chất cũng là task

Nhờ `clone()`, Linux tạo ra các task có mức chia sẻ tài nguyên khác nhau. Vì vậy process và thread chỉ khác nhau ở mức độ share resource và grouping semantics.

### 17.6. Scheduler schedule task, không schedule khái niệm mơ hồ

Mỗi runnable entity đều có cost. Càng nhiều thread/task, lock contention và context switch càng có thể đắt.

### 17.7. Trong Embedded Linux, mọi thứ này càng quan trọng hơn

Vì tài nguyên hạn chế, latency nhạy cảm và hệ thống phải chạy ổn định lâu dài.

## Học tiếp theo hướng nào?

Nếu muốn đào sâu tiếp sau bài này, thứ tự rất hợp lý là:

1. `execve()` và cách kernel nạp ELF
2. `mm_struct`, page table, TLB, page fault
3. `clone()` flags và `pthread_create()`
4. CFS internals, `vruntime`, runqueue
5. RT scheduling: `SCHED_FIFO`, `SCHED_RR`, priority inversion
6. kernel stack, syscall entry, trap frame
7. interrupt context, softirq, workqueue, kthread
8. debug scheduler bằng `perf sched` và `ftrace`
