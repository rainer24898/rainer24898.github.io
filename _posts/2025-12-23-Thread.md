## 1. Thread là gì?

**Thread** là đơn vị thực thi nhỏ nhất mà scheduler của hệ điều hành có thể lập lịch chạy.

Có thể hiểu ngắn gọn:

* **Process** là một chương trình đang chạy cùng không gian tài nguyên riêng.
* **Thread** là một luồng thực thi bên trong process đó.

Một process có thể có:

* 1 thread duy nhất → single-threaded
* nhiều thread → multi-threaded

Mỗi thread có:

* program counter riêng
* register set riêng
* stack riêng
* trạng thái scheduling riêng

Nhưng các thread trong cùng một process thường **chia sẻ**:

* cùng virtual address space
* vùng code/text
* global variables
* heap
* file descriptor table
* signal dispositions (đa phần ở mức process)

### Hình dung nhanh

```text
Process A
├── Thread 1  -> stack riêng, registers riêng
├── Thread 2  -> stack riêng, registers riêng
└── Thread 3  -> stack riêng, registers riêng

Shared inside Process A:
- code
- data
- heap
- mapped libraries
- file descriptors
```

Thread giúp một process làm nhiều việc “song song logic” cùng lúc, đặc biệt hữu ích khi:

* chờ I/O
* xử lý nhiều connection
* pipeline công việc
* tận dụng nhiều CPU core

## 2. Vì sao cần thread?

### 2.1 Tăng độ phản hồi

Ví dụ một ứng dụng:

* thread UI giữ giao diện mượt
* thread worker tải dữ liệu
* thread log ghi file nền

Nếu chỉ có một luồng, khi đọc file lớn hoặc gọi network blocking, toàn bộ ứng dụng có thể bị treo cảm giác.

### 2.2 Tận dụng đa lõi CPU

Trên máy nhiều core:

* nhiều thread có thể chạy thật sự song song trên nhiều core
* thích hợp cho workload CPU-bound

### 2.3 Thiết kế tách nhiệm vụ rõ ràng

Ví dụ server mạng:

* thread accept connection
* thread pool xử lý request
* thread monitor health
* thread flush statistic

### 2.4 Overhead nhỏ hơn process

So với process, thread thường:

* tạo nhanh hơn
* hủy nhanh hơn
* switch rẻ hơn tương đối
* giao tiếp với nhau dễ hơn vì dùng chung memory

Tuy nhiên, đổi lại thread làm tăng nguy cơ lỗi khó debug như:

* race condition
* deadlock
* use-after-free liên thread
* memory visibility bug

## 3. Process vs Thread

## 3.1 So sánh nhanh

| Tiêu chí           | Process  | Thread                        |
| ------------------ | -------- | ----------------------------- |
| Đơn vị tài nguyên  | Có       | Không hoàn toàn               |
| Đơn vị thực thi    | Có       | Có                            |
| Không gian địa chỉ | Riêng    | Dùng chung trong cùng process |
| Stack              | Riêng    | Riêng                         |
| Global/heap        | Riêng    | Dùng chung trong cùng process |
| Tạo/hủy            | Nặng hơn | Nhẹ hơn                       |
| IPC / giao tiếp    | Khó hơn  | Dễ hơn qua shared memory      |
| Cô lập lỗi         | Tốt hơn  | Kém hơn                       |

## 3.2 Điểm khác nhau quan trọng nhất

* **Process** tách biệt tài nguyên mạnh hơn.
* **Thread** chia sẻ nhiều tài nguyên nên phối hợp nhanh hơn nhưng nguy hiểm hơn.

Ví dụ:

* một thread ghi bậy vào con trỏ toàn cục → có thể làm hỏng cả process
* một process khác crash thường không trực tiếp làm hỏng memory của process bạn

## 4. Mô hình bộ nhớ của thread

Trong một process đa luồng, memory layout có thể hình dung như sau:

```text
High Address
+---------------------------+
| Thread 3 stack            |
+---------------------------+
| Thread 2 stack            |
+---------------------------+
| Thread 1 stack            |
+---------------------------+
| mmap regions / shared lib |
+---------------------------+
| Heap (shared)             |
+---------------------------+
| BSS / Data (shared)       |
+---------------------------+
| Text / Code (shared)      |
+---------------------------+
Low Address
```

### Shared giữa các thread

* biến global
* biến static
* heap do `malloc()` cấp phát
* vùng `mmap()`
* file descriptor

### Riêng cho từng thread

* stack frame
* biến local tự động trong hàm
* register context
* thread ID
* signal mask riêng
* thread-local storage riêng

### Ví dụ cực kỳ quan trọng

```c
int g_counter = 0;   // shared

void *worker(void *arg)
{
    int local = 0;   // private của từng thread
    local++;
    g_counter++;     // shared, dễ race nếu không khóa
    return NULL;
}
```

Ở đây:

* `local` nằm trên stack riêng của từng thread
* `g_counter` nằm vùng data/global, tất cả thread cùng nhìn thấy

## 5. Thread trong Linux nhìn từ kernel

Đây là phần rất quan trọng nếu bạn học theo hướng hệ điều hành hoặc debug Linux sâu.

## 5.1 Linux không phân biệt quá cứng process và thread

Trong Linux kernel, cả process và thread đều được biểu diễn dưới dạng **task**.

Cấu trúc trung tâm là:

* `task_struct`

Mỗi thực thể được scheduler lập lịch chạy là một `task_struct`.

### Ý nghĩa thực tế

* process chính là một task
* thread cũng là một task
* điểm khác nhau nằm ở mức **chia sẻ tài nguyên**

## 5.2 clone() là nền tảng tạo thread

Trong Linux, thread user-space tạo qua `pthread_create()` nhưng bên dưới thường quy về syscall kiểu `clone()` hoặc biến thể mới tương đương.

`clone()` cho phép chỉ định những gì được chia sẻ:

* `CLONE_VM` → chia sẻ address space
* `CLONE_FILES` → chia sẻ file descriptor table
* `CLONE_SIGHAND` → chia sẻ signal handlers
* `CLONE_THREAD` → cùng thread group
* `CLONE_FS` → chia sẻ thông tin filesystem context

Nếu chia sẻ nhiều tài nguyên, ta có mô hình giống “thread”.
Nếu không chia sẻ, nó gần hơn với “process”.

## 5.3 TID và TGID

Trong Linux:

* **TID**: thread ID ở mức kernel
* **TGID**: thread group ID, thường là PID của main thread

Trong cùng một process đa luồng:

* mỗi thread có TID khác nhau
* cả nhóm có thể cùng TGID

### Hình dung

```text
Process / Thread group (TGID = 1000)
├── Main thread   TID = 1000
├── Worker A      TID = 1001
└── Worker B      TID = 1002
```

## 5.4 Scheduler nhìn thread như thế nào?

Scheduler không thực sự quan tâm “đây là process hay thread” theo cách người học ban đầu thường nghĩ.
Nó chọn **task nào runnable** thì cấp CPU cho task đó.

Nghĩa là scheduler lập lịch ở mức **thread/task**, không phải mức “ứng dụng tổng thể”.

## 5.5 Context của mỗi thread

Mỗi thread có ít nhất các thành phần riêng để có thể resume lại việc chạy:

* kernel stack riêng
* saved registers
* scheduling state
* priority / policy
* CPU affinity
* pending state liên quan

Đây là lý do dù cùng chạy chung code, kernel vẫn biết chính xác **thread nào đang chạy**, đang block ở đâu, thuộc process nào.

## 6. Tạo và quản lý thread bằng POSIX pthread

Trên Linux user-space, API phổ biến nhất là **POSIX Threads** (`pthread`).

Cần include:

```c
#include <pthread.h>
```

Khi build thường dùng:

```bash
gcc main.c -pthread -o app
```

## 6.1 Tạo thread với `pthread_create`

Khai báo:

```c
int pthread_create(pthread_t *thread,
                   const pthread_attr_t *attr,
                   void *(*start_routine)(void *),
                   void *arg);
```

Ý nghĩa:

* `thread`: nơi nhận ID của thread
* `attr`: thuộc tính thread, có thể `NULL`
* `start_routine`: hàm thread bắt đầu chạy
* `arg`: tham số truyền vào

### Ví dụ cơ bản

```c
#include <stdio.h>
#include <pthread.h>

void *worker(void *arg)
{
    const char *name = (const char *)arg;
    printf("Hello from thread: %s\n", name);
    return NULL;
}

int main(void)
{
    pthread_t tid;

    pthread_create(&tid, NULL, worker, "T1");
    pthread_join(tid, NULL);
    return 0;
}
```

## 6.2 Kết thúc thread

Một thread có thể kết thúc khi:

* hàm `start_routine` return
* gọi `pthread_exit()`
* bị cancel
* cả process kết thúc

Ví dụ:

```c
pthread_exit(NULL);
```

## 6.3 Chờ thread kết thúc với `pthread_join`

```c
int pthread_join(pthread_t thread, void **retval);
```

Dùng để:

* đợi thread kết thúc
* thu giá trị trả về của thread
* thu hồi tài nguyên của thread joinable

Nếu không join một thread joinable, có thể phát sinh dạng “rò tài nguyên thread” ở mức runtime.

## 7. Joinable vs Detached

## 7.1 Joinable thread

Mặc định, thread thường là **joinable**.

Nghĩa là:

* khi thread kết thúc, tài nguyên chưa được giải phóng hoàn toàn ngay
* thread khác cần `pthread_join()` để reap/thu hồi

## 7.2 Detached thread

Detached nghĩa là:

* không cần join
* khi kết thúc thì hệ thống tự thu hồi tài nguyên thread

Tạo detached bằng:

```c
pthread_detach(tid);
```

Hoặc cấu hình qua `pthread_attr_t`.

## 7.3 Khi nào dùng?

* **joinable**: khi cần lấy kết quả, cần biết lúc nào thread xong
* **detached**: fire-and-forget, không cần chờ kết quả

Lưu ý: detached thread khó quản lý vòng đời hơn nếu nó dùng dữ liệu mà main thread có thể giải phóng sớm.

## 8. Race condition và vì sao thread dễ lỗi

**Race condition** xảy ra khi kết quả chương trình phụ thuộc vào thứ tự chạy xen kẽ giữa các thread.

Ví dụ kinh điển:

```c
counter = counter + 1;
```

Dòng này nhìn có vẻ đơn giản nhưng thực tế có thể gồm nhiều bước:

1. load `counter`
2. cộng thêm 1
3. store trở lại

Nếu hai thread cùng làm mà không khóa:

```text
Initial counter = 0

Thread A: load 0
Thread B: load 0
Thread A: store 1
Thread B: store 1

Kết quả cuối = 1, nhưng mong đợi = 2
```

Đây gọi là **lost update**.

### Ví dụ C có race

```c
#include <stdio.h>
#include <pthread.h>

int counter = 0;

void *worker(void *arg)
{
    for (int i = 0; i < 100000; i++) {
        counter++;
    }
    return NULL;
}

int main(void)
{
    pthread_t t1, t2;

    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);

    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    printf("counter = %d\n", counter);
    return 0;
}
```

Kết quả có thể không phải 200000.

## 9. Các cơ chế đồng bộ quan trọng

Đây là phần cốt lõi khi làm việc với thread.

## 9.1 Mutex

**Mutex** dùng để bảo vệ critical section sao cho tại một thời điểm chỉ một thread được vào.

### Mẫu dùng

```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&lock);
/* critical section */
pthread_mutex_unlock(&lock);
```

### Sửa race counter bằng mutex

```c
#include <stdio.h>
#include <pthread.h>

int counter = 0;
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void *worker(void *arg)
{
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&lock);
        counter++;
        pthread_mutex_unlock(&lock);
    }
    return NULL;
}

int main(void)
{
    pthread_t t1, t2;

    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);

    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    printf("counter = %d\n", counter);
    pthread_mutex_destroy(&lock);
    return 0;
}
```

## 9.2 Spinlock

Spinlock cũng khóa vùng tới hạn, nhưng thay vì ngủ chờ thì thread **busy wait**.

Phù hợp khi:

* critical section rất ngắn
* không muốn context switch nặng
* low-level/kernel hoặc môi trường latency nhạy cảm

Không phù hợp nếu lock giữ lâu vì đốt CPU.

## 9.3 Read-Write Lock

Cho phép:

* nhiều reader cùng lúc
* writer phải độc quyền

Phù hợp hệ thống **đọc nhiều, ghi ít**.

## 9.4 Semaphore

Semaphore là bộ đếm đồng bộ.

Có thể dùng để:

* giới hạn số lượng tài nguyên đồng thời
* báo hiệu giữa các thread

### Ví dụ ý tưởng

* producer tạo item → `sem_post`
* consumer chờ item → `sem_wait`

## 9.5 Condition Variable

Condition variable cho phép thread **ngủ chờ một điều kiện logic** thay vì polling.

Rất hay dùng với mutex theo mẫu producer-consumer.

### Mẫu chuẩn

```c
pthread_mutex_lock(&lock);
while (!condition) {
    pthread_cond_wait(&cond, &lock);
}
/* xử lý khi condition đúng */
pthread_mutex_unlock(&lock);
```

Tại sao dùng `while` chứ không `if`?

* để chống spurious wakeup
* để kiểm tra lại điều kiện sau khi được đánh thức

## 9.6 Atomic operation

Với biến nhỏ và thao tác đơn giản, có thể dùng atomic thay cho mutex trong một số bài toán.

Ví dụ C11 atomics:

```c
#include <stdatomic.h>

atomic_int counter;
atomic_fetch_add(&counter, 1);
```

Atomic không thay thế toàn bộ mutex.
Nó phù hợp cho thao tác nhỏ, rõ ràng, không phải cho mọi critical section phức tạp.

## 10. Deadlock, livelock, starvation

## 10.1 Deadlock

Deadlock là tình huống hai hoặc nhiều thread chờ nhau vô hạn.

### Ví dụ

* Thread A giữ lock 1, chờ lock 2
* Thread B giữ lock 2, chờ lock 1

```text
A: lock(L1) ------ wait L2
B: lock(L2) ------ wait L1
```

Cả hai đứng mãi.

### 4 điều kiện kinh điển dễ dẫn đến deadlock

1. mutual exclusion
2. hold and wait
3. no preemption
4. circular wait

### Cách giảm nguy cơ

* áp đặt thứ tự khóa cố định
* tránh nested locking khi không cần
* dùng trylock có timeout/rollback logic
* giảm số lock phải giữ cùng lúc

## 10.2 Livelock

Các thread không bị block cứng, nhưng liên tục né nhau mà không tiến triển.

## 10.3 Starvation

Một thread bị thiếu cơ hội chạy hoặc thiếu cơ hội lấy lock trong thời gian dài.

Ví dụ:

* thread priority thấp luôn bị thread ưu tiên cao lấn át
* writer starvation khi reader liên tục vào RW lock

## 11. Scheduler, context switch và hiệu năng

## 11.1 Scheduler là gì?

Scheduler quyết định:

* thread nào được chạy
* chạy trên CPU nào
* trong bao lâu
* theo policy nào

## 11.2 Thread state thường gặp

Khái niệm mức cao:

* running
* runnable
* blocked / sleeping
* terminated

### Hình dung

```text
create -> runnable -> running -> blocked -> runnable -> running -> exit
```

## 11.3 Context switch là gì?

Khi CPU chuyển từ chạy thread A sang thread B, kernel phải lưu/phục hồi context.

Có thể bao gồm:

* general registers
* stack pointer
* program counter
* scheduling metadata
* đôi khi tác động đến cache/TLB locality

### Hệ quả

Context switch không miễn phí.
Nhiều thread quá mức có thể làm hiệu năng tệ đi vì:

* switch quá nhiều
* cache miss tăng
* tranh chấp lock tăng
* false sharing

## 11.4 Oversubscription

Nếu có quá nhiều thread runnable so với số core:

* scheduler phải chia thời gian dày đặc
* tăng overhead scheduling
* throughput có thể giảm

Ví dụ máy 8 core mà tạo 500 thread CPU-bound thường không hiệu quả.

## 11.5 CPU-bound vs I/O-bound

* **CPU-bound**: thread chủ yếu tính toán → số thread thường nên gần số core hợp lý
* **I/O-bound**: thread hay block chờ I/O → có thể cần nhiều thread hơn số core

## 12. Thread-local storage (TLS)

Có những dữ liệu ta muốn “dùng global-like” nhưng thực ra mỗi thread phải có một bản riêng.

Đó là **thread-local storage**.

### Ví dụ C11

```c
_Thread_local int tls_counter = 0;
```

Hoặc compiler extension:

```c
__thread int tls_counter;
```

Mỗi thread nhìn thấy biến riêng của mình.

### Dùng khi nào?

* buffer tạm riêng cho mỗi thread
* error state riêng
* cache nhỏ riêng theo thread
* tránh lock cho dữ liệu độc lập

### Lưu ý

TLS không giải quyết việc chia sẻ dữ liệu giữa thread; nó là kỹ thuật để **không cần chia sẻ** trong một số trường hợp.

## 13. Signal và thread

Signal trong chương trình đa luồng là phần nhiều người học dễ nhầm.

## 13.1 Một số điểm cần nhớ

* signal disposition thường ở mức process
* signal mask có thể riêng theo từng thread
* một signal gửi tới process có thể được deliver cho một thread phù hợp
* signal gửi đích danh thread có thể dùng cơ chế riêng như `pthread_kill`

## 13.2 Vì sao nguy hiểm?

Nếu handler signal làm những việc không async-signal-safe như:

* `printf()`
* `malloc()`
* lock mutex

thì rất dễ gây lỗi khó lường.

### Quy tắc thực dụng

Trong signal handler:

* làm ít nhất có thể
* thường chỉ set cờ kiểu `sig_atomic_t`
* xử lý chính để main loop/thread khác làm

## 14. Một số lỗi thực chiến rất hay gặp

## 14.1 Truyền địa chỉ biến local cho thread rồi hàm cha kết thúc

Ví dụ lỗi:

```c
void create_thread_bad(void)
{
    int value = 10;
    pthread_t tid;
    pthread_create(&tid, NULL, worker, &value);
}
```

Nếu thread dùng `&value` sau khi hàm kết thúc, con trỏ trở thành không an toàn.

## 14.2 Truyền cùng một biến loop cho nhiều thread

```c
for (int i = 0; i < 5; i++) {
    pthread_create(&t[i], NULL, worker, &i);
}
```

Tất cả thread có thể đọc cùng một địa chỉ `&i`, dẫn tới giá trị sai.

Cách đúng: cấp vùng nhớ riêng cho từng thread hoặc dùng mảng tham số riêng.

## 14.3 Free dữ liệu khi thread khác còn dùng

Rất hay gặp trong chương trình nhiều callback, queue, worker thread.

## 14.4 Double lock hoặc quên unlock

Ví dụ đường lỗi return sớm mà bỏ sót `pthread_mutex_unlock()`.

## 14.5 Dùng `volatile` để cố “chống race”

`volatile` **không phải** công cụ đồng bộ thread.
Nó không thay thế mutex, semaphore hay atomic memory ordering.

## 14.6 False sharing

Hai thread sửa hai biến khác nhau, nhưng hai biến nằm cùng cache line → hiệu năng tụt mạnh do cache coherence traffic.

Ví dụ:

```c
struct Stats {
    long a;
    long b;
};
```

Nếu thread 1 cập nhật `a`, thread 2 cập nhật `b`, vẫn có thể đụng nhau ở mức cache line.

### Cách giảm

* padding / align cache line
* tách dữ liệu nóng theo thread
* gom batch rồi merge

## 16. Khi nào nên dùng thread, khi nào không?

## 16.1 Nên dùng thread khi

* cần tận dụng nhiều core
* cần xử lý nhiều tác vụ tương đối độc lập
* cần responsive trong khi có tác vụ blocking
* có thể thiết kế ownership dữ liệu rõ ràng

## 16.2 Không nên lạm dụng thread khi

* bài toán quá nhỏ
* dữ liệu chia sẻ phức tạp, lock chằng chịt
* overhead đồng bộ lớn hơn lợi ích
* có thể dùng event loop / async model phù hợp hơn

## 16.3 Một nguyên tắc thực chiến

Thread không làm chương trình nhanh lên một cách tự động.

Nó chỉ giúp khi:

* workload thật sự song song được
* bottleneck không nằm ở lock / memory / I/O serialized
* thiết kế tránh contention tốt

## 17. Ví dụ C hoàn chỉnh

Dưới đây là ví dụ **producer-consumer** đơn giản bằng `pthread`, `mutex`, `condition variable`.
Nó minh họa gần như đủ các ý cốt lõi:

* nhiều thread
* dữ liệu shared
* đồng bộ bằng lock
* chờ điều kiện đúng bằng condvar
* shutdown sạch

```c
#include <stdio.h>
#include <stdlib.h>
#include <pthread.h>
#include <unistd.h>

#define BUFFER_SIZE 8
#define PRODUCE_COUNT 20

typedef struct {
    int data[BUFFER_SIZE];
    int head;
    int tail;
    int count;
    int done;

    pthread_mutex_t lock;
    pthread_cond_t not_empty;
    pthread_cond_t not_full;
} queue_t;

static void queue_init(queue_t *q)
{
    q->head = 0;
    q->tail = 0;
    q->count = 0;
    q->done = 0;
    pthread_mutex_init(&q->lock, NULL);
    pthread_cond_init(&q->not_empty, NULL);
    pthread_cond_init(&q->not_full, NULL);
}

static void queue_destroy(queue_t *q)
{
    pthread_mutex_destroy(&q->lock);
    pthread_cond_destroy(&q->not_empty);
    pthread_cond_destroy(&q->not_full);
}

static void queue_push(queue_t *q, int value)
{
    pthread_mutex_lock(&q->lock);

    while (q->count == BUFFER_SIZE) {
        pthread_cond_wait(&q->not_full, &q->lock);
    }

    q->data[q->tail] = value;
    q->tail = (q->tail + 1) % BUFFER_SIZE;
    q->count++;

    pthread_cond_signal(&q->not_empty);
    pthread_mutex_unlock(&q->lock);
}

static int queue_pop(queue_t *q, int *value)
{
    pthread_mutex_lock(&q->lock);

    while (q->count == 0 && !q->done) {
        pthread_cond_wait(&q->not_empty, &q->lock);
    }

    if (q->count == 0 && q->done) {
        pthread_mutex_unlock(&q->lock);
        return 0;
    }

    *value = q->data[q->head];
    q->head = (q->head + 1) % BUFFER_SIZE;
    q->count--;

    pthread_cond_signal(&q->not_full);
    pthread_mutex_unlock(&q->lock);
    return 1;
}

static void *producer(void *arg)
{
    queue_t *q = (queue_t *)arg;

    for (int i = 1; i <= PRODUCE_COUNT; i++) {
        queue_push(q, i);
        printf("[producer] pushed %d\n", i);
        usleep(50000);
    }

    pthread_mutex_lock(&q->lock);
    q->done = 1;
    pthread_cond_broadcast(&q->not_empty);
    pthread_mutex_unlock(&q->lock);

    return NULL;
}

static void *consumer(void *arg)
{
    queue_t *q = (queue_t *)arg;
    int value;

    while (queue_pop(q, &value)) {
        printf("[consumer] got %d\n", value);
        usleep(120000);
    }

    return NULL;
}

int main(void)
{
    pthread_t prod_tid;
    pthread_t cons_tid;
    queue_t q;

    queue_init(&q);

    if (pthread_create(&prod_tid, NULL, producer, &q) != 0) {
        perror("pthread_create producer");
        queue_destroy(&q);
        return 1;
    }

    if (pthread_create(&cons_tid, NULL, consumer, &q) != 0) {
        perror("pthread_create consumer");
        pthread_join(prod_tid, NULL);
        queue_destroy(&q);
        return 1;
    }

    pthread_join(prod_tid, NULL);
    pthread_join(cons_tid, NULL);

    queue_destroy(&q);
    return 0;
}
```

### Build

```bash
gcc thread_demo.c -pthread -O2 -g -o thread_demo
```

### Những gì ví dụ này minh họa

* queue là dữ liệu shared
* mutex bảo vệ tính nhất quán của queue
* producer chờ khi buffer đầy
* consumer chờ khi buffer rỗng
* `while` được dùng đúng chuẩn quanh `pthread_cond_wait`
* shutdown có cờ `done` để thoát sạch

## 18. Tóm tắt các ý quan trọng

* Thread là đơn vị thực thi; process là đơn vị tài nguyên lớn hơn.
* Các thread trong cùng process chia sẻ memory và tài nguyên nên giao tiếp nhanh nhưng dễ lỗi.
* Mỗi thread có stack, registers, scheduling context riêng.
* Trên Linux, thread và process đều là `task`; thread thường được tạo dựa trên cơ chế `clone()` với cờ chia sẻ tài nguyên.
* `pthread_create`, `pthread_join`, mutex, condvar, semaphore là bộ công cụ nền tảng.
* Race condition là lỗi phổ biến nhất khi dữ liệu shared bị truy cập không đồng bộ.
* `volatile` không phải công cụ chống race.
* Dùng lock đúng chỗ, ownership rõ ràng, hạn chế shared mutable state là nguyên tắc sống còn.
* Nhiều thread hơn không đồng nghĩa nhanh hơn; phải xét scheduler, core count, contention, cache locality.
* Debug thread cần nhìn cả backtrace tất cả thread, trạng thái block, lock ordering và lifetime của dữ liệu.

## Kết luận

Thread là một chủ đề rất “đắt giá” trong C/Linux vì nó nằm đúng giao điểm giữa:

* lập trình hệ thống
* hiệu năng
* tính đúng đắn
* debugging thực chiến

Học thread tốt không chỉ là nhớ `pthread_create()` hay `pthread_mutex_lock()`, mà là hiểu sâu các câu hỏi sau:

* dữ liệu nào shared, dữ liệu nào private?
* ownership nằm ở đâu?
* thread có thể chạy xen kẽ theo những cách nào?
* điều kiện wake/sleep đã đúng chưa?
* shutdown path có sạch không?
* có deadlock/race/lifetime bug không?

Khi nắm chắc các ý này, bạn sẽ đọc code đa luồng tốt hơn rất nhiều, debug core dump chính xác hơn, và viết phần mềm hệ thống ổn định hơn.
