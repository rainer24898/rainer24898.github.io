# FAPI 5G Ultra-Technical Notes  
## Góc nhìn của engineer 5G/L1: state, timing, wire-format, parser và debug thực chiến

> Bản này được viết theo tinh thần **ultra-technical**: ưu tiên góc nhìn **implement / integrate / trace / debug**.  
> Nội dung vẫn giữ **toàn bộ hình ảnh gốc**, nhưng diễn giải theo hướng gần với công việc của engineer làm **MAC-PHY interface, L1 adapter, PDU packing, timing correlation và interoperability**.

---

## 0. Cách đọc tài liệu này

Đừng đọc FAPI như một danh sách message. Hãy đọc nó như một hệ thống gồm **4 lớp ràng buộc** chạy đồng thời:

```text
(1) state contract
(2) time contract
(3) structural contract
(4) semantic contract
```

Nếu 1 trong 4 lớp trên sai, hệ thống có thể:

- không crash,
- vẫn gửi/nhận message,
- nhưng PHY và MAC đã không còn “nói cùng một ngôn ngữ”.

Đó là lý do bug FAPI thường khó hơn bug C thông thường.

## 1. FAPI thực sự nằm ở đâu trong kiến trúc

FAPI nằm ở ranh giới nơi **scheduler intent** được biến thành **PHY executable work**.

```text
+------------------------------------------------------+
| MAC scheduler / HARQ / link adaptation / RLC triggers|
+------------------------------------------------------+
                       |
                       |  FAPI P5 / P7
                       v
+------------------------------------------------------+
| L1 control plane / PDU dispatcher / PHY executor     |
+------------------------------------------------------+
                       |
                       v
+------------------------------------------------------+
| baseband processing / coding / mapping / RF control  |
+------------------------------------------------------+
```

Nói cực ngắn:

- **MAC nghĩ theo policy**
- **PHY nghĩ theo exact execution**
- **FAPI là biên dịch giữa policy và execution**

## 2. FAPI, nFAPI, P19: boundary nào dành cho cái gì

### 2.1. FAPI
Dành cho MAC ↔ PHY semantics ở mức implementation.

### 2.2. nFAPI
Khi boundary MAC–PHY cần “đi qua mạng”, bạn cần một biến thể transportable hơn.

### 2.3. P19
Đi xuống FEU / RF / digital frontend control. Nó không thay P5/P7; nó mở rộng chuỗi điều khiển xuống thấp hơn.

### 2.4. Cách nhớ ngắn

```text
P5/P7  = điều khiển + runtime giữa MAC và PHY
nFAPI  = mang cùng semantics đó qua network split
P19    = kéo đường control xuống frontend/RF
```

## 4. State machine: không phải tài liệu minh họa, mà là contract runtime

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.002.png)

**Hình 2 – State machine diagram**

### 4.1. Minimal state set

```text
IDLE -> CONFIGURED -> RUNNING
```

### 4.2. Nghĩ theo invariant

#### IDLE invariant
- không xử lý slot runtime
- có thể query capability
- chưa consume P7

#### CONFIGURED invariant
- cấu hình hợp lệ đã được commit
- chưa consume runtime data path như khi RUNNING

#### RUNNING invariant
- P7 timing đang có hiệu lực
- slot acceptance window đã bắt đầu có ý nghĩa
- sai order/sai timing sẽ không còn là “warning” mà là functional error

### 4.3. Hình logic bổ trợ

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.003.png)
![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.004.png)

**Hình 3–4 – State logic triển khai**

### 4.4. Mẫu code state gate

```c
typedef enum {
    FAPI_PHY_IDLE = 0,
    FAPI_PHY_CONFIGURED = 1,
    FAPI_PHY_RUNNING = 2
} fapi_phy_state_e;

typedef enum {
    FAPI_MSG_PARAM_REQ  = 0x0100,
    FAPI_MSG_CONFIG_REQ = 0x0101,
    FAPI_MSG_START_REQ  = 0x0102,
    FAPI_MSG_STOP_REQ   = 0x0103,
    FAPI_MSG_RESET_REQ  = 0x0104
} fapi_msg_id_e;

static int fapi_state_accepts(fapi_phy_state_e st, uint16_t msg_id)
{
    switch (st) {
    case FAPI_PHY_IDLE:
        return (msg_id == FAPI_MSG_PARAM_REQ ||
                msg_id == FAPI_MSG_CONFIG_REQ ||
                msg_id == FAPI_MSG_RESET_REQ);

    case FAPI_PHY_CONFIGURED:
        return (msg_id == FAPI_MSG_PARAM_REQ ||
                msg_id == FAPI_MSG_START_REQ ||
                msg_id == FAPI_MSG_RESET_REQ);

    case FAPI_PHY_RUNNING:
        return (msg_id == FAPI_MSG_STOP_REQ  ||
                msg_id == FAPI_MSG_RESET_REQ);

    default:
        return 0;
    }
}
```

Điều quan trọng là:

> state validation nên nằm ở **cổng nhận message**, không phải đợi xuống sâu mới fail.

## 5. Message wire-format: phải tách bạch wire layout và in-memory layout

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.005.png)

**Hình 5 – Simplified FAPI message format**

Một message FAPI cần được nhìn dưới 3 lớp:

```text
[wire bytes] -> [parser view] -> [typed internal representation]
```

Không nên gộp 3 lớp này thành 1 bằng cách cast buffer thẳng sang struct.

### 5.1. Vì sao cast thẳng nguy hiểm

- alignment có thể sai,
- compiler padding có thể chen byte,
- endian có thể khác,
- flexible-length list không thể map an toàn bằng struct tĩnh.

### 5.2. Quy tắc production-grade

```text
wire buffer        : uint8_t *
cursor/offset      : size_t
scalar read/write  : helper rd_u16/rd_u32/wr_u16/wr_u32
lists              : parse theo count/length
typed objects      : tạo ra sau khi validate xong
```

## 7. Bit mapping: nơi rất nhiều bug “silent” xuất hiện

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.007.png)

**Hình 7 – Payload mapping by bit significance**

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.008.png)

**Hình 8 – Payload mapping by bit position**

### 7.1. 4 thứ phải tách bạch hoàn toàn

```text
bit significance in spec
byte order on wire
CPU endianness
debugger memory display order
```

### 7.2. Ví dụ sai kinh điển

Bạn có field dài 11 bit. Spec bảo bit 10 là MSB logic. Bạn pack bằng cách:

```c
uint16_t x = field & 0x07ff;
memcpy(buf, &x, 2);
```

Code này **có thể đúng trên máy local**, nhưng vẫn sai theo spec vì:

- spec định nghĩa on-wire order,
- `memcpy` phản chiếu RAM layout của CPU,
- không chắc trùng rule của wire protocol.

### 7.3. Mẫu pack/unpack explicit

```c
static void put_11bits_msb_first(uint8_t *buf, uint16_t v)
{
    v &= 0x07ff;
    buf[0] = (uint8_t)(v >> 3);          /* bits 10..3 */
    buf[1] = (uint8_t)((v & 0x7) << 5);  /* bits 2..0 at top */
}
```

Điều đáng chú ý:

- code dài hơn,
- nhưng explicit,
- review được,
- trace bằng hex dump được.

## 8. TLV: simple in concept, dangerous in implementation

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.009.png)

**Hình 9 – Ví dụ format của `PARAM.response`**

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.010.png)

**Hình 10 – Cấu trúc TLV**

![](/assets/img/post/FAPI/Aspose.Words.d989787b-4774-47d7-998d-63b4e1ce3463.011.png)

**Hình 11 – Ý tưởng dùng union để tối ưu bộ nhớ**

### 8.1. TLV contract

Mỗi TLV cần đảm bảo ít nhất:

- tag hợp lệ,
- length hợp lệ,
- value nằm trọn trong message boundary,
- count tổng không vượt message length,
- không có integer overflow khi cộng offset.

### 8.2. Tagged-union pattern nên dùng

```c
typedef enum {
    FAPI_TLV_KIND_NONE = 0,
    FAPI_TLV_KIND_U8,
    FAPI_TLV_KIND_U16,
    FAPI_TLV_KIND_U32,
    FAPI_TLV_KIND_BUF
} fapi_tlv_kind_e;

typedef struct {
    uint16_t tag;
    uint16_t length;
    fapi_tlv_kind_e kind;
    union {
        uint8_t u8;
        uint16_t u16;
        uint32_t u32;
        struct {
            const uint8_t *ptr;
            uint16_t len;
        } buf;
    } val;
} fapi_tlv_t;
```

### 8.3. Vì sao phải có `kind`

Không có `kind`, union chỉ là vùng nhớ chồng lấp. Bạn sẽ không biết:

- lúc này `val.u32` có hợp lệ không,
- hay thực ra phải đọc `val.buf`.

## 13. Struct, union và parser design trong C

### 13.1. Wire header

```c
#pragma pack(push, 1)
typedef struct {
    uint16_t msg_id;
    uint16_t msg_len;
} fapi_wire_hdr_t;
#pragma pack(pop)
```

### 13.2. Parse view thay vì object ownership

```c
typedef struct {
    uint16_t msg_id;
    uint16_t msg_len;
    const uint8_t *body;
} fapi_msg_view_t;
```

`view` rất hữu ích vì:

- không copy dữ liệu,
- không ép allocate sớm,
- cho phép validate trước rồi mới materialize object sau.

### 13.3. Safe decode helpers

```c
static uint16_t fapi_rd_u16_le(const uint8_t *p)
{
    return (uint16_t)p[0] | ((uint16_t)p[1] << 8);
}

static int fapi_parse_hdr(const uint8_t *buf, size_t len, fapi_msg_view_t *out)
{
    if (!buf || !out) return -1;
    if (len < 4) return -2;

    out->msg_id  = fapi_rd_u16_le(buf + 0);
    out->msg_len = fapi_rd_u16_le(buf + 2);

    if ((size_t)out->msg_len + 4 > len)
        return -3;

    out->body = buf + 4;
    return 0;
}
```

### 13.4. TLV cursor parser

```c
typedef struct {
    const uint8_t *buf;
    size_t len;
    size_t off;
} fapi_cursor_t;

typedef struct {
    uint16_t tag;
    uint16_t length;
    const uint8_t *value;
} fapi_tlv_view_t;

static int fapi_cursor_next_tlv(fapi_cursor_t *c, fapi_tlv_view_t *tlv)
{
    if (!c || !tlv) return -1;
    if (c->off + 4 > c->len) return -2;

    tlv->tag    = fapi_rd_u16_le(c->buf + c->off);
    tlv->length = fapi_rd_u16_le(c->buf + c->off + 2);
    c->off += 4;

    if (c->off + tlv->length > c->len)
        return -3;

    tlv->value = c->buf + c->off;
    c->off += tlv->length;
    return 0;
}
```

### 13.5. Tagged union cho PDU type

```c
typedef enum {
    FAPI_PDU_DCI = 1,
    FAPI_PDU_PDSCH,
    FAPI_PDU_PUSCH,
    FAPI_PDU_PUCCH,
    FAPI_PDU_PRACH,
    FAPI_PDU_SRS
} fapi_pdu_type_e;

typedef struct {
    uint16_t rnti;
    uint8_t mcs;
    uint8_t harq_id;
} fapi_pdsch_pdu_t;

typedef struct {
    uint16_t rnti;
    uint8_t mod_order;
    uint8_t rv_index;
} fapi_pusch_pdu_t;

typedef struct {
    fapi_pdu_type_e type;
    union {
        fapi_pdsch_pdu_t pdsch;
        fapi_pusch_pdu_t pusch;
    } u;
} fapi_any_pdu_t;
```

### 13.6. Sai lầm phải tránh

#### Sai lầm 1: serialize trực tiếp struct local
```c
send(fd, &local_struct, sizeof(local_struct), 0);
```

#### Sai lầm 2: giả định `sizeof(struct)` == message length theo spec
Không chắc đúng nếu:
- compiler padding khác,
- field optional khác,
- list variable-length khác.

#### Sai lầm 3: đọc union nhầm nhánh
Nếu `type == FAPI_PDU_PDSCH` mà code vẫn đọc `u.pusch`, bug có thể không crash nhưng semantic sai hoàn toàn.

## 14. ERROR.indication: đừng chỉ log, hãy phân loại

Trong hệ thống thật, `ERROR.indication` nên được phân loại ít nhất theo 4 nhóm:

```text
STATE error
TIMING error
STRUCTURE error
SEMANTIC error
```

Ví dụ:

- **STATE**: START khi chưa CONFIG
- **TIMING**: request đến muộn
- **STRUCTURE**: length/TLV/count mismatch
- **SEMANTIC**: PDU fields mâu thuẫn logic hoặc vượt capability

Khi log theo 4 nhóm này, RCA sẽ rõ hơn rất nhiều.

## 16. Checklist review code FAPI cho C/C++

### 16.1. Wire-format
- Có parse từ byte buffer không?
- Có helper endian riêng không?
- Có check integer overflow khi tính offset không?

### 16.2. State
- Có validate state ở entry point không?
- Có reject sai state sớm không?

### 16.3. Timing
- Có log arrival lag không?
- Có reject late message theo rule rõ ràng không?

### 16.4. Union/TLV
- Có `type`/`kind` đi kèm union không?
- Có validate `length` trước khi dereference value không?

### 16.5. Debuggability
- Có log đủ `msg_id/sfn/slot/pdu_idx` không?
- Có thể dump raw bytes khi cần RCA không?

## 17. Kết luận

FAPI 5G không khó nhất ở chỗ số lượng message nhiều. Nó khó ở chỗ một implement đúng phải đồng thời đúng ở cả 4 mặt:

```text
đúng state
đúng timing
đúng structure
đúng semantics
```

Đó là lý do cùng một bug MAC–PHY có thể biểu hiện thành:

- missing indication,
- CRC fail,
- throughput drop,
- interop mismatch,
- hoặc chỉ là một `ERROR.indication` rất ngắn ngủi.

Muốn debug tốt FAPI, đừng chỉ đọc spec theo chiều “message nghĩa là gì”.  
Hãy đọc theo chiều:

```text
message này được chấp nhận khi nào?
nó ràng buộc state nào?
nó ràng buộc slot nào?
nó bind với payload/PDU nào?
nếu fail, fail theo lớp nào?
```

Khi tư duy như vậy, bạn sẽ thấy FAPI không còn là “danh sách API”, mà là **hợp đồng thực thi giữa scheduler và PHY**.
